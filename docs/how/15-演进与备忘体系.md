## 15.1 三大决策记录体系

`writer-agent` 用三套互补的体系记录「我们做了什么决策 + 为什么 + 修了什么 bug」:

| 体系 | 路径 | 何时写 | 读者 |
| ---- | ---- | ------ | ---- |
| **技术备忘** | `技术难点与解决方案备忘/01-17/*.md` | 选型时一次性记录「为什么这样设计」 | 新人 / 重构者 |
| **OpenSpec 变更** | `openspec/changes/{change-id}/` | 每次大变更,先 proposal → apply → archive | 审计 / 协作者 |
| **Bug 记录** | `docs/bugs/{N}-{slug}.md` | 修复 bug 时,锚点 `per Bug N` 留在 commit message | 维护者 / 用户 |

三者互相引用:`OpenSpec change` 引用 `备忘` 提供选型理由,`bug 修复 commit` 引用 `Bug N` 锚点。

### 备忘的「生命周期」

- **形成**:遇到选型问题,写备忘,记录候选方案 + 理由(不立即实装)
- **应用**:OpenSpec change 引用备忘决策,apply 时落地
- **演进**:备忘需要更新时(如 LangGraph 改 Engine 事件流),增量修订并加「更新日期」标记

## 15.3 OpenSpec 变更管理

> 对应代码:`openspec/` 目录

### OpenSpec 工作流

```
1. /opsx:explore       # 探索模式:思考问题,不写代码
2. /opsx:propose       # 提案模式:写 proposal + design + tasks + specs
3. openspec validate --strict  # 校验提案完整性
4. /opsx:apply         # 应用模式:按 tasks 实现
5. openspec sync specs # 把 delta spec 同步到主 spec
6. /opsx:archive       # 归档到 openspec/changes/archive/
```

### 提案结构

```
openspec/changes/<change-id>/
├── proposal.md          # 一句话改动 + 决策点
├── design.md            # 详细设计 + 备选 + 权衡
├── tasks.md             # 拆分成可勾选任务
└── specs/               # delta spec
    └── <spec-name>/spec.md
```

## 15.4 Bug 记录 `docs/bugs/`

> 对应代码:`docs/bugs/{N}-{slug}.md`

每个 bug 修复对应一篇文档,**commit message 引用锚点 `per Bug N`**。

### 当前 5 个 Bug

```
docs/bugs/
├── README.md
├── 01-tool-loop-not-rebound.md      # Bug 01:set_project_root 没重建 tool_loop
├── 02-action-answer-ignored-by-tool-loop.md   # Bug 02:ReActAgent 收不到 directive body
├── 03-review-chapter-always-deterministic.md  # Bug 03:review_chapter 总走 fallback
├── 04-whitelist-vs-first-segment.md  # Bug 04:写入白名单首段匹配越界
└── 05-workflow-module-globals.md     # Bug 05:workflow 模块级全局变量导致交叉污染
```

### Bug 文档结构

```markdown
# Bug {N}: {一句话标题}

## 现象
{用户视角的症状}

## 根因
{技术根因 + 触发条件}

## 修复
{代码 diff + commit hash}

## 回归测试
{新增的 test case,确保不再发生}

## 防御
{Protocol / 文档 / 工具的强化,避免类似 bug}
```

### 锚点约定

```bash
git commit -m "fix(engine): 重建 tool_loop on set_project_root (per Bug 01)"
```

锚点 `per Bug 01` 让 `git log --grep="Bug 01"` 立刻找到对应文档。

## 15.5 三大体系的协同示例

以 `chg-add-write-edit-glob` 为例:

1. **备忘 13-核心Tool设计.md** 记录「Tool 必须命名 kw-only」「写入必须白名单」等决策
2. **OpenSpec change** `chg-add-write-edit-glob/proposal.md` 写「补齐 3 个写入/编辑/glob 工具」
3. **apply** 时 commit:`fix(tool): safe_write_file AGENT.md 3-stage guard (per Bug 04)`
4. **Bug 04 文档** `docs/bugs/04-whitelist-vs-first-segment.md` 记录「祖先路径前缀匹配」修复细节

四个文档互相引用,形成完整的「选型 → 提案 → 实装 → bug 修复」闭环。

## 15.7 给新加入者的「3 步上手」

1. **读 README + CLAUDE.md** —— 知道项目做什么、怎么跑
2. **读 `docs/how/01-总览与四层架构.md`** —— 建立心智地图
3. **按需读 `技术难点与解决方案备忘/`** —— 理解每个决策的「为什么」

**不要直接读代码**:代码里有大量「per 备忘 N」「per Bug M」「per chg-X」锚点,直接读会迷失。

## 15.8 给维护者的「3 步修改」

1. **加新 SKILL.md / agent .md** → 无需改 Python,reload 即生效
2. **加新 builtin Tool** → 写 `tools/builtin/*.py` + 在 `built_tool_registry` 注册,顺便补备忘 13
3. **大重构** → 走 OpenSpec:`/opsx:explore` → `/opsx:propose` → `openspec validate` → `/opsx:apply` → `/opsx:archive`
