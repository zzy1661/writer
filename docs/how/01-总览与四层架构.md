## 1.1 一句话定义

`writer-agent` 是一个面向中文用户的**长篇小说写作 CLI**(命令行工具),目标辅助用户完成 **20–50 万字小说**的构思、规划、分章、正文生成与修订。

它不是「一个聊天框调用 GPT」,而是一个**有状态、有项目结构、有状态机校验、有安全 Tool 边界**的多层架构:

- 输入是一行行 REPL 命令(`/大纲` `/目录` `/创作` `/审核` `/init` / 自然语言)
- 引擎把它转成结构化 `AgentAction`,再派发给 LLM / Tool / Workflow
- 每轮产出**事件流**(TextChunk / ActionEvent / ToolCall / ToolResult / Interrupt / Done / ErrorEvent),REPL 据此渲染
- 项目落到磁盘:每个小说是一个独立目录(`AGENT.md` + `outline/` + `manuscript/` + `characters/` + `world/` + `notes/` + `创意/` + `.writer/`)

---

## 1.2 为什么用「四层」?

设计原则是 **Claude Code §十·最小接口 DI**(参考备忘 16):

> 每层只向下依赖,**绝不跨层穿透**;层与层之间只通过 Protocol + AsyncGenerator 事件流交互。

| 跨层反模式 | 我们怎么做 |
| ---------- | ---------- |
| CLI 直接调 LLM | CLI 只 `async for event in run_engine(...)`,不知道 LLM 存在 |
| Engine 直接 new Tool | Engine 只看到 `ToolRegistry` Protocol,具体实现可换 |
| Workflow 直接读环境变量 | Workflow 通过 `EngineContext` + `EngineDeps` 注入 |
| Agent 直接写文件 | Agent 通过 Tool 协议,Tool 经 Runtime 白名单校验 |
| LLM Provider 直接调 prompt 模板 | Provider 只暴露 `BaseChatModel`,prompt 在 Engine / Loop 层拼 |

## 1.5 关键设计约束(全仓库通用)

写代码前必须理解的 6 条:

1. **`IntentRouter` 是 Protocol,不是具体类**。Engine 只依赖 Protocol;`RuleBasedIntentRouter` 是无网络 MVP,`LlmIntentRouter` (LangChain structured output,双 provider)与 `CompositeRouter` (rule-first + LLM fallback)已实装并由 `production_deps()` 默认装配。

2. **`EngineDeps` 是 DI 边界**。Engine 不直接 `new` 协作者,所有外部依赖通过 Protocol 注入;`production_deps(...)` 是默认装配;`rebind_*` 系列方法用于 `EngineSession.set_project_root` 时的热替换。

3. **Engine 输出是事件流**。`TextChunk` / `ActionEvent` / `ToolCall` / `ToolResult` / `Interrupt` / `Done` / `ErrorEvent` 全部 `@dataclass(frozen=True)`。`AgentAction` 是 **Pydantic `BaseModel` + `frozen=True`**,**不是 dataclass** —— 用 `model_copy(update=...)` 而不是 `dataclasses.replace(...)`。

4. **REPL 路由原则**:除框架命令(`/退出` `/帮助` `/状态`)外,斜杠命令与自然语言**一律交给 engine**,CLI 层不重复维护命令路由。

5. **Tool 安全**:所有 builtin Tool 必须经 `runtime.safe_path()` 防越界;写入类(`safe_write_file` / `safe_edit_file`)额外走 `runtime.allowed_write_paths` 白名单 + AGENT.md 3-stage guard;Tool 必须用**命名 keyword 参数**(`def run(self, runtime, *, path: str)`),不能用 `**kwargs`,否则 LangChain `args_schema` 无法生成。

6. **业务规则是 directive**:SKILL.md frontmatter(`command` / `description` / `requires_states` / `body` / `references` / `scripts`)驱动多个下游表面(`/帮助` / Tab 补全 / 状态机拦截 / Engine 分发);加新 shipped directive 或项目级 `.md` 覆盖**不需要改 CLI / engine 代码**。

## 1.6 心智地图:一次完整的 REPL turn

以 `> /大纲 一个穿越到唐朝的程序员` 为例,完整路径如下:

```
[1] prompt_toolkit 读入一行
    │
[2] cli/main.py:handle_repl_input() 判断
    │   · 不是 EXIT / HELP / /状态 / /init flag 形式
    │   · 也不是 S0 时的 /init(纯 <name>),所以交给 engine
    ▼
[3] EngineSession.compose_pending_input(user_input, session.pending_interrupt)
    │   · 没有 pending interrupt → 直接返回原 input
    ▼
[4] async for event in run_engine(ctx, deps, config):
    │   · ctx = EngineContext(user_input, project_root, project_state, session_id)
    │   · deps = session.deps (EngineDeps Protocol)
    │   · config = EngineConfig(...)
    ▼
[5] deps.route(user_input, project_state) → AgentAction
    │   · RuleBasedIntentRouter 命中前缀 "/大纲"
    │   · 产出 action_type="run_command", command="/大纲"
    ▼
[6] validate_command_available("/大纲", ...) → 检查矩阵
    │   · directive_registry.get("/大纲") 命中,requires_states 满足
    ▼
[7] directive = deps.directive_registry.get("/大纲")
    │   · 命中 _shipped/大纲/SKILL.md
    │   · 加载 body + references(4-act-template.md / examples.md)
    ▼
[8] async for event in _run_directive(directive, ctx, deps, cfg):
    │   · resolve_references(body, references)
    │   · 构造 agent_action = AgentAction(action_type="answer_directly", command="/大纲", answer=directive.body)
    │   · async for event in deps.tool_loop.run(agent_action, ctx, deps, cfg):
    ▼
[9] LLMToolLoop.run():
    │   · _initial_messages(): 拼 system prompt = base + directive body + agent identity + router hint
    │   · while tool_calls_made < MAX_LOOP_STEPS:
    │       ├─ ai_message = _invoke_model(messages)
    │       ├─ parsed = _parse_ai_message(ai_message)
    │       ├─ if parsed.action_type == "call_tool":
    │       │     · yield ToolCall(...)
    │       │     · result = registry.invoke(tool_name, runtime, **arguments)
    │       │     · yield ToolResult(...)
    │       │     · 继续下一轮 LLM 调用
    │       └─ elif parsed.action_type == "answer_directly":
    │             · yield TextChunk(answer)
    │             · yield Done("answered")
    ▼
[10] CLI _render_event() 逐个消费事件
     · TextChunk → console.print(text, markup=False, highlight=False)
     · ToolCall → console.print(f"[tool] {name}({args})")
     · ToolResult → console.print(f"[result] {output}")
     · Done → console.print(f"[done] {reason}")
```

## 1.7 数据流:从用户输入到落盘文件

```
用户输入 "/大纲 穿越到唐朝"
   ↓
AgentAction(command="/大纲", action_type="run_command")
   ↓ (engine 分派)
SkillDirective(command="/大纲", body="...四幕模板...", references={4-act-template.md, examples.md})
   ↓ (LLM 读 directive body)
LLM 思考:我需要先了解项目当前状态,然后写 outline/大纲.md
   ↓ (LLM 产出 AgentAction(call_tool, tool_name="safe_read_file", arguments={"path": "AGENT.md"}))
ToolCall(name="safe_read_file", arguments={"path": "AGENT.md"})
   ↓ (ToolRegistry.invoke 经 ToolRuntime.safe_path() 校验)
ToolResult(output="项目: 长安程序员 / 题材: 其他 / 当前状态: S1 初始化...")
   ↓ (LLM 继续思考,产出 safe_write_file 调用)
ToolCall(name="safe_write_file", arguments={"path": "outline/大纲.md", "content": "..."})
   ↓ (ToolRegistry.invoke 走 8 个白名单校验:允许写入 outline/)
ToolResult(output="已写入 outline/大纲.md")
   ↓ (LLM 产出 answer_directly)
TextChunk("大纲已生成,共 N 段...")
Done(reason="answered", payload={...})
```

落盘的文件:
- `<project_root>/outline/大纲.md` —— 实际大纲(由 safe_write_file 写入)
- `<project_root>/AGENT.md` —— 状态字段 `当前状态` 可能更新(由 guard 守护)
