### 2.1 层间边界

| 层 | 对外接口 | 不应关心 |
| --- | --- | --- |
| L1 CLI | 命令字符串 / 事件回调 | LLM 类型、agent 状态机 |
| L2 会话控制 | `Engine(runner=Runner(deps, cfg))` API + `RunnerContext` (frozen) | Runner 内部 dispatch |
| L3 引擎核心 | `Runner(deps, cfg).run(ctx) -> AsyncIterator[Event]`（新代码入口；不保留 compat shim） | 具体 LLM 模型名、Tool 实现细节 |
| L4 Provider | 标准化 `ChatRequest` / `ChatResponse` / `BaseChatModel` | 上下文、角色、prompt 模板 |

每层只向下依赖,不跨层穿透。例如 L1 永远不直连 L4;L2 永远不 import L3 内部;L3 永远不知道 LLM 是不是 OpenAI。

## 三、Runner 事件流 + IntentRouter 协议(原 §三 替代)

### 3.1 为什么从 LangGraph 迁回 Engine / Runner 事件流

2026-07-01 的旧设计把 LangGraph 状态图作为 L3 的核心:

```text
plan_outline → write_chapter → proofread → [history_check] → review_gate
```

但**前置依赖**(LLM router、Tool 层、checkpoint 协议)在 2026-07-03 都没到位;真正落地时,Engine AsyncGenerator + `match` 单层 dispatch 已经在跑 MVP,而 LangGraph 一直没接进生产代码。

2026-07-03 重构后:

- **前台调度** = `IntentRouter` Protocol(MVP=`RuleBasedIntentRouter`,未来=`LlmIntentRouter`,生产=`CompositeRouter`)
- **后台执行**(单轮派发) = `Runner` 类(`src/writer/runner/runner.py`,**2026-07-16 由 `Engine` 重命名**)持有 `RunnerDeps` + `RunnerConfig`,公开 `run(ctx)` AsyncGenerator;内部 `Runner._engine_loop()` 5 路 `match` dispatch,辅助 helper `Runner._run_tool` / `_run_tool_loop` / `_run_workflow` / `_run_agent` / `_run_directive` 均为私有方法
- **会话容器**(跨轮主控) = `Engine` 类(`src/writer/session/engine.py`,**2026-07-16 由 `EngineSession` 重命名**)持 `Runner` 实例 + 跨 turn 状态(`session_id` / `project_root` / `pending_interrupt` / `turns`);`Engine.set_project_root` 通过 `Runner.replace_deps(new_deps)` 整体替换 deps
- **长任务** = `_DefaultRunnerDeps.run_workflow(name, ctx)` 委派到 `WORKFLOWS = {"write_chapter": write_chapter_run, "review_chapter": review_chapter_run}`;两个工作流均为 LangGraph 5 节点图(2026-07-09 PR2/PR3 实装)

### 3.2 双轴映射:阶段 vs 角色

旧文档中的"阶段(Planner/Outliner/Writer/Reviewer)"与"角色(编剧/历史/校对)"对照表依然适用,但**只在 LangGraph 工作流**里表达。当前的 Runner 形态下,角色由 `AgentAction.role` 字段携带(`story_agent` / `proofreader` / `historian` / `reviewer`),阶段隐含在 action_type 里(`run_command` / `call_tool` / `start_workflow`)。

| action_type | 角色(role 字段) | 引擎分支 |
| --- | --- | --- |
| `answer_directly` | (任意) | `yield Done("answered")` |
| `run_command` | 由 directive / 命令决定 | `_run_init_command` / `_run_directive` / `Done("command_pending")` |
| `call_tool` | `story_agent` (默认) | `_run_tool` (rule-only) / `_run_tool_loop` (LLM) |
| `start_workflow` | `story_agent` / `reviewer` | `_run_workflow` → `Done("workflow_completed")`(2026-07-09 PR2 实装 LangGraph 后) |
| `ask_user` | (任意) | `yield Interrupt` + `Done("ask_user")` |
| `kind="agent"` | 优先派发(per `fea-agent-mirror`) | `_run_agent` → LLM 走 agent body + agent identity |

### 3.3 Runner 输入契约(原 §3.3 替代)

```python
from dataclasses import dataclass
from pathlib import Path


@dataclass(frozen=True)
class RunnerContext:
    """当次 turn 的全部输入,函数参数。LLM / LangGraph 任何外部输入都只能写进这个。"""

    user_input: str
    project_root: Path | None = None
    project_state: str = "S0"
    session_id: str = ""
```

`RunnerConfig`(frozen,见 `src/writer/runner/config.py`)是当次 turn 的运行时配置(`session_id` / `fast_mode` / 等),消费者在 AsyncGenerator 流式消费事件时,config 不会突变,可放心依赖。

### 3.4 事件层级

```python
DoneReason = Literal[
    "answered", "command_pending", "tool_pending",
    "ask_user", "aborted", "tool_completed", "tool_loop_completed",
    "workflow_completed",          # 2026-07-09 新增:LangGraph write_chapter 跑完
    # 注意:"workflow_pending" 不再是合法 DoneReason,pending 分支走 aborted + decision
]


@dataclass(frozen=True)
class Event: ...


@dataclass(frozen=True)
class TextChunk(Event):
    text: str


@dataclass(frozen=True)
class ActionEvent(Event):
    action: AgentAction


@dataclass(frozen=True)
class ToolCall(Event):
    name: str
    arguments: dict[str, Any]


@dataclass(frozen=True)
class ToolResult(Event):
    name: str
    output: str


@dataclass(frozen=True)
class Interrupt(Event):
    type: Literal["choice", "text", "confirm"]
    prompt: str
    options: list[str] | None = None


@dataclass(frozen=True)
class Done(Event):
    reason: DoneReason
    payload: dict[str, Any] | None = None


@dataclass(frozen=True)
class ErrorEvent(Event):
    message: str
    traceback: str | None = None
```

每个 turn 产出一个终结 `Done`,REPL 据此渲染。

- `tool_loop_completed` — 2026-07-08 新增,`ReActAgent` 走完 `MAX_LOOP_STEPS=5` 优雅耗尽(与 `aborted` 区分)。
- `workflow_completed` — 2026-07-09 新增,LangGraph `write_chapter` 图跑完后的终结事件(`Runner._run_workflow` 把 `WorkflowResult.status="completed"` 映射到此 reason)。`workflow_pending` 不再是合法 reason —— pending 分支走 `aborted + decision` 表达。

### 3.6 节点契约(LangGraph 2026-07-09 PR2 实装)

每个 LangGraph 节点对外契约(以 `write_chapter` 为例,代码 `src/writer/workflows/write_chapter.py`):

- **输入**:`WriterState` TypedDict(部分字段)
- **副作用**:写文件到 `草稿/chapter-<chapter_id>.md`,原子更新 `chapter_summaries.json`(`persist_outputs` 节点);LLM 调 LLM(`plan_chapter` 强制 real + `draft_chapter` / `review_gate` real 模式)或纯确定性节点(`prep_context` / `proofread` / `persist_outputs`)
- **输出**:更新后的 `WriterState` 字典(部分字段)

`build_writer_graph()` 编译的图(`src/writer/workflows/write_chapter.py::build_writer_graph`)拓扑:

```
prep_context → plan_chapter → draft_chapter → proofread
                                          ↓
                                    review_gate
                                    /        \
                              (rewrite)    (end)
                                  ↓           ↓
                            draft_chapter   persist_outputs → END
```

`REVIEW_THRESHOLD = 7`,`max_retries = 2`,循环最多跑 `max_retries + 1` 次。

### 3.7 检查点(2026-07-09 实装,2026-07-14 修订,2026-07-16 重命名同步)

**LangGraph 自带 checkpointer**:`write_chapter._build_checkpointer` 优先用 `SqliteSaver`(落到 `<project_root>/.writer/checkpoints.sqlite`),无 `project_root` 或 `SqliteSaver` 不可用时降级到 `MemorySaver`。`thread_id` 用 `session_id` 或 `f"write-{chapter_id}"`,让同一 chapter_id 跨 REPL 轮次可恢复。

**会话控制层**:`Engine`(L2,长寿命,2026-07-16 由 `EngineSession` 重命名)持有跨 turn 状态,与 LangGraph checkpointer 共存:

- `session_id` frozen UUID,跨 turn 唯一
- `runner: Runner` 一次性构建(`__post_init__` 装配 `RunnerDeps` 后包装 `Runner`)
- `turns: list[TurnRecord]` 追加式 turn 历史(`turn_index` / `user_input` / `done_reason` / `timestamp`)
- `pending_interrupt: Interrupt | None` 上轮未消费的中断
- `set_project_root()` 触发 `ToolRuntime` / `DirectiveRegistry` / `AgentRegistry` 三件套热替换,最终通过 `Runner.replace_deps(new_deps)` 整体替换 `Engine` 实例内部的 runner 的 deps

两层职责清晰:LangGraph 维护单次工作流内部 checkpoint;`Engine` 维护跨 turn 会话状态;`Runner` 是每轮 AsyncGenerator 入口(短命)。

| `foreshadow_search` | `id`/`tags`/`status`/`chapter_range`/`keyword`(5 kw-only) | 查询 `<project_root>/伏笔.yaml`;多条件 AND;S0 走 sentinel 识别 |
| `chapter_locate` | `chapter=None` | S0 stub,echo 输入 |
| `wordcount` | `text=None`, `path=None` | 中文友好字数统计 |

**写入路径白名单**(per `chg-add-write-edit-glob` D2):`{manuscript, outline, characters, world, notes, 创意, .writer/cache, .writer/agents}`,挂在 `ToolRuntime.allowed_write_paths`(None = 默认白名单);AGENT.md 通过 guard 显式旁路;backups 落到 `.writer/backups/<relpath>.<ISO-timestamp>`(默认开启)。

完整规范见 [备忘 13](../技术难点与解决方案备忘/13-核心Tool设计.md)。

### 4.3 IO 错误处理(2026-07-08 修订)

`PermissionError` / `OSError` / `UnicodeDecodeError` 在 builtin Tool 内转 `ToolResult(metadata.error=...)`,**不**外溢到 engine 的 `except Exception`。这条约定保证 REPL turn 不会被单条 Tool 调用失败打断。

### 4.4 Tool 安全约束

- 所有 builtin Tool 必须经 `ToolRuntime.safe_path()` 防越界
- Tool 必须用**命名 keyword 参数**(`def run(self, runtime, *, path: str)`),不能用 `**kwargs`
- `ToolRegistry.register()` 名字重复立即抛 `ValueError`
- `ToolRegistry.describe()` 2026-07-08 新增,`ToolDescriptor` 提供给 LLM 工具循环的工具目录
- `writer.tools.langchain_bridge.to_langchain_tools(registry, runtime)` 提供 LC `BaseTool` 包装

## 五、LLM Provider(L4 层,2026-07-08 修订)

### 5.1 双 provider 路由

| 路径 | 何时用 | 实现位置 |
| --- | --- | --- |
| **native `bind_tools` / `with_structured_output`** | OpenAI 兼容协议原生支持 structured output(deepseek-v3 / GPT / 等) | `LlmIntentRouter.__init__` 检测 `needs_json_prompt_structured_output(settings) is False` |
| **JSON-prompt fallback** | 模型不支持原生 structured output(DeepSeek 部分版本) | `LlmIntentRouter._use_json_prompt=True` 分支;`writer.llm.structured.invoke_structured_json` 渲染 Pydantic schema 进 prompt,要求模型输出 JSON,后端解析回 `AgentAction` |

`needs_json_prompt_structured_output(settings)` 是切换条件;`production_deps` 装配 `LlmIntentRouter` 时决定走哪条。

### 5.2 `ReActAgent` ReAct 多步循环

`src/writer/llm/agent.py::ReActAgent` 实现 ReAct 风格多步工具调用:

- 预算 `MAX_LOOP_STEPS=5`
- `_parse_ai_message` 解析顺序:`AIMessage.tool_calls` → `additional_kwargs._json_action` → content JSON → 最后兜底为 `answer_directly`(prose 内容也算 answer)
- `ToolError` 从循环内向上传播,让 engine 外层 `except ToolError` 兜底 → `ErrorEvent + Done(aborted)`
- 预算耗尽:`TextChunk` 兜底文本 + `Done(tool_loop_completed, payload={tool_calls_made, last_output})` —— **优雅耗尽**,与 `aborted` 区分

### 5.3 L4 接口契约

L3 唯一看到的接口是 LangChain `BaseChatModel`,具体模型名字 L3 不感知。`Settings`(`src/writer/config/settings.py`)通过 `WRITER_*` 环境变量切换 Provider:

```bash
WRITER_MODEL=deepseek-v3
WRITER_BASE_URL=https://api.deepseek.com/v1
WRITER_API_KEY=...
```

未配置 API key 时 `settings.has_api_key is False`,`production_deps` 跳过 `ReActAgent` 装配,engine 走同步 `_run_tool` 路径(`tool_completed`)。

### 5.4 错误恢复

| 错误 | 处理 | 兜底 |
| --- | --- | --- |
| `LLM_TIMEOUT` / 5xx / 网络 | `ReActAgent` 内指数退避 3 次 | 单步 fail → `ToolError` 上抛 → engine `except ToolError` → `ErrorEvent + Done(aborted)` |
| `LLM_OUTPUT_TRUNCATED` | **不重试**,转人工 | `ToolResult(truncated=True, metadata=...)` 透传,LLM 决定下一步 |
| 上下文超限 | `ReActAgent` 当前未做主动压缩;交给 LLM 模型 | 顶层 `except Exception` 兜底 |
| `ToolError` | 上抛 → engine `except ToolError` → `ErrorEvent + Done(aborted)` | 引擎边界 `except Exception` 兜底 |

## 六、错误处理 / 可观测性(2026-07-08 修订)

### 6.1 错误分类与策略

| 错误类型 | 来源 | 处理 | 用户感知 |
| --- | --- | --- | --- |
| `ToolError` | builtin Tool 业务异常(path / permission / schema / IO) | engine `except ToolError` | `ErrorEvent + Done(aborted)`,`traceback` 字段带 formatted stack |
| `SkillError` | SKILL.md directive body 抛出的领域异常 | engine `except SkillError` | `ErrorEvent + Done(aborted, payload={'command':...})` |
| `Exception` | 任何意外 | engine `except Exception` 兜底 | `ErrorEvent + Done(aborted)` |
| LLM flaky | `CompositeRouter.fallback` 抛异常 | 自动回退 `primary(规则版)` | 不阻断 REPL,`log.warning` 留痕 |
| LLM 预算耗尽 | `ReActAgent.MAX_LOOP_STEPS=5` | 落 `TextChunk` 兜底文本 + `Done(tool_loop_completed)` | 不算 failed |

### 6.2 ErrorEvent.traceback(2026-07-05 M4 修复后)

`ErrorEvent.message` 是人读摘要;`ErrorEvent.traceback` 携带 formatted stack trace,REPL 输出可直接粘贴进 bug report,无需重跑 + debugger。

### 6.3 日志

- **开发模式**(`WRITER_LOG=debug`):全量 debug 流到 stderr
- **生产模式**(默认):只有 INFO/WARN/ERROR 写到 `~/.writer/logs/<date>.log`
- **结构化日志**:JSON Lines,字段 `ts / level / layer / node / event / project / chapter`
- **日志轮转**:单日 > 50MB 自动压缩,保留 7 天

### 6.4 可观测性

- **不接入外部 APM**(本地 CLI,无意义)
- L1 输出"任务时间线"面板:每个 turn 起止时间、token 消耗、模型档位
- REPL turn 末尾自动汇总:`耗时 Xmin / Done.reason / 调档次数 / 重试次数`
