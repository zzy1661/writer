## 一、设计概要

| 决策项 | 选择 | 替代 |
| --- | --- | --- |
| 架构层次 | 4 层(CLI / 会话控制 / Agent 核心 / LLM Provider) | 单体、3 层、5 层 |
| Agent 核心模型 | 多 Agent 状态图(LangGraph) | 单一调度 + LCEL、动态 Agent 树(AutoGen 风格) |
| LangChain 角色 | 仅 RAG / 记忆层(LLM 包装、嵌入、检索器、分块器、Prompt 模板) | 全 LangChain 抽象、纯 LangGraph+原生 SDK |
| LangGraph 嵌入 | 双模式:默认 embedded + 本地 SQLite;`--dev` 启动 Studio | 永远独立 dev server、永远纯本地 |
| LLM Provider | OpenAI 兼容协议 + 三档模型路由(high/mid/low) | 单一模型、按 Provider 自动选型 |
| RAG | FAISS-CPU + BM25 + 中文嵌入(bge / m3e) | Pinecone、纯向量、纯 BM25 |

**核心思想**:LangGraph 主导流程编排,LangChain 提供"读"的工程能力,LLM Provider 是可替换的薄壳。

### 2.1 层间边界

| 层 | 对外接口 | 不应关心 |
| --- | --- | --- |
| L1 CLI | 命令字符串 / 事件回调 | LLM 类型、会话状态 |
| L2 会话控制 | 状态机查询 / checkpoint API | 角色细节、prompt 内容 |
| L3 Agent | 任务描述字符串、最终输出 | Provider 细节、checkpoint 实现 |
| L4 Provider | 标准化 `ChatRequest` / `ChatResponse` | 上下文、角色、prompt 模板 |

每层只向下依赖,不跨层穿透——例如 L1 永远不直连 L4。

## 三、LangGraph 状态图 + 角色双轴映射

### 3.1 核心图(简化)

```
                ┌────────────────────┐
                │  plan_outline 节点  │ ← Outliner prompt (编剧顾问)
                └─────────┬──────────┘
                          │ 角色=编剧顾问,模型=高端
                          ▼
                ┌────────────────────┐
        ┌───────┤  write_chapter 节点 │ ← Writer prompt (编剧顾问)
        │       └─────────┬────────┘
        │                 │ 角色=编剧顾问,模型=中端
        │                 ▼
        │       ┌────────────────────┐
        │       │ proofread 节点     │ ← Proofreader prompt (校对)
        │       └─────────┬────────┘
        │                 │ 角色=校对,模型=轻量
        │                 │ 条件边:is_historical → 进入 history_check
        │                 ▼
        │       ┌────────────────────┐
        │       │ history_check 节点  │ ← Historian prompt (历史顾问,仅历史题材)
        │       └─────────┬────────┘
        │                 │ 角色=历史顾问,模型=高端
        │                 ▼
        │       ┌────────────────────┐
        └───────┤ review_gate 节点    │ ← Reviewer prompt (三角色综合)
                └─────────┬──────────┘
                          │ 角色=reviewer,模型=中端
                          ▼
                    [END / 续写]
```

### 3.2 双轴映射(关键)

CLAUDE.md 的 4 角色(Planner / Outliner / Writer / Reviewer)= **流程阶段**(状态图节点)  
DESIGN.md 的 3 子代理(编剧顾问 / 历史顾问 / 校对)= **执行角色**(调用 LLM 用的 prompt 模板)

| 阶段 \ 角色 | 编剧顾问 | 历史顾问 | 校对 | Reviewer(综合) |
| --- | :---: | :---: | :---: | :---: |
| plan_outline | ✓ | — | — | — |
| write_chapter | ✓ | — | — | — |
| proofread | — | — | ✓ | — |
| history_check(仅历史题材) | — | ✓ | — | — |
| review_gate | — | — | — | ✓(汇聚) |

**作用**:
- 阶段(节点)决定**流程顺序**与**何时/怎样流转**
- 角色决定**调用 LLM 时用哪个 prompt 模板 + 哪些工具 + 什么上下文**

### 3.3 WriterState 状态定义

```python
from typing import Literal, TypedDict

class ReviewReport(TypedDict):
    word_count_ok: bool
    proofread_issues: list[str]
    history_issues: list[str]
    consistency_issues: list[str]
    reader_perspective: list[str]
    needs_rewrite: bool

class WriterState(TypedDict):
    project_root: str
    chapter_id: str            # "1.3"
    stage: Literal[
        "plan_outline",    # 章纲生成(首次/变更后 章节前)
        "write",           # 写章节正文
        "proofread", "history_check", "review"
    ]
    role: Literal["writer", "proofreader", "historian", "reviewer"]
    canon: dict                # RAG 检索:大纲/人物/世界观/伏笔相关片段
    context: dict              # 当前章节 + 前后各 1-2 节摘要
    draft: str                 # 当前草稿
    review: ReviewReport       # review_gate 汇聚的结果
    feedback: list[str]        # 修订建议
    retry_count: int           # 整章重试计数
    is_historical: bool        # 历史题材标志
    history: list[dict]        # 消息历史(LangGraph 内部)
```

### 3.4 节点契约

每个节点对外契约:
- **输入**:`state` 字典(部分字段)
- **副作用**:写文件到 `正文草稿/` 或 `大纲/大纲.md` 修订记录
- **输出**:更新后的 `state` 字典(部分字段)

### 3.5 关键边

| 边 | 条件 | 目标 |
| --- | --- | --- |
| write_chapter → proofread | 无条件 | proofread |
| proofread → history_check | `state["is_historical"] is True` | history_check |
| proofread → review_gate | `state["is_historical"] is False` | review_gate |
| history_check → review_gate | 无条件 | review_gate |
| review_gate → write_chapter | `state["review"]["needs_rewrite"]` and `state["retry_count"] < 3` | write_chapter |
| review_gate → END | 其它 | END |

### 3.6 检查点

- 每个节点结束都写 SQLite checkpoint(`langgraph-checkpoint` ≥ 2.0 库)
- 中断恢复:CLI 启动时检测当前项目存在未完结的会话,提示「上一会话在 X 节点中断,从这里继续?」
- 用户 `Ctrl+C` 强制 flush 当前 checkpoint

## 四、LangChain 子系统(RAG / 记忆)

LangChain 在 L3 里**只**负责"读与检索",**不**拥有 Agent 状态机。

### 4.1 子系统边界

```
┌─────────────── L3 Agent 核心 ──────────────────────────────┐
│                                                             │
│   状态图节点 (Outliner / Writer / ...)                     │
│        │ 调用                                              │
│        ▼                                                    │
│   ┌──── 共享层 (LangChain 域) ──────────────────────────┐   │
│   │ RAG 检索器 ── 读 canon + 历史章节摘要               │   │
│   │ 嵌入模型    ── 国产 bge / m3e                      │   │
│   │ 向量库      ── FAISS(本地,每项目独立索引)          │   │
│   │ 分块器      ── 中文友好(标点优先)                  │   │
│   │ Prompt 模板 ── 按角色分目录加载                     │   │
│   └────────────────────────────────────────────────────┘   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 4.2 检索目标(每个项目建索引)

| 索引 | 来源 | 分块策略 | 何时用 |
| --- | --- | --- | --- |
| `canon` | `大纲/大纲.md` | 按"卷/章"语义分块,保留层级 | WriterState.canon |
| `personas` | `人物/*.md` | 每个角色一个 chunk | WriterState.canon |
| `world` | `世界观/*.md` | 按子主题分块 | WriterState.canon |
| `foreshadow` | `伏笔/伏笔库.md` | 每条伏笔一个 chunk | WriterState.canon |
| `history` | `史实/*.md` | 按事件/年份 | WriterState.canon(仅历史) |
| `past_chapters` | `正文/` + `正文草稿/` | 段落级摘要,不喂原文 | WriterState.context |

### 4.3 检索 API

```python
class CanonSlice(TypedDict):
    outline_chunks: list[str]
    persona_chunks: list[str]
    world_chunks: list[str]
    foreshadow_chunks: list[str]
    history_chunks: list[str]

class RagRetriever:
    def query_canon(self, chapter_id: str, k: int = 8) -> CanonSlice: ...
    def query_history(self, chapter_id: str, k: int = 5) -> list[ChapterSummary]: ...
    def check_consistency(self, new_text: str, anchor: str = "personas") -> list[Conflict]: ...
```

### 4.4 记忆分层

| 记忆层 | 存储 | 用途 | 生命周期 |
| --- | --- | --- | --- |
| L0 短期 | LangGraph SQLite checkpoint | 当前 state.history(消息) | 单会话,中断可恢复 |
| L1 项目级 RAG | FAISS(`~/.writer/projects/<sha>/faiss/`) | canon / personas / world / foreshadow / past_chapters | 随项目存在 |
| L2 用户偏好 | `~/.writer/prefs.toml` | 模型偏好、风格偏好、API key | 跨项目 |

**不引入 LangChain Memory 类**(已废弃,与 LangGraph checkpoint 重叠)。

### 4.5 嵌入 / 检索配置

- **嵌入**:BAAI/bge-large-zh-v1.5(开源、中文 SOTA),可通过 `WRITER_EMBEDDING_MODEL` 覆盖
- **向量库**:FAISS-CPU(本地足够,无需 Pinecone / Qdrant)
- **检索**:向量 + 关键词混合(FAISS + BM25;BM25 用 `jieba` 中文分词)
- **摘要**:已写字用 LLM 跑过 **段落级摘要**(`chapter_summaries.json`),只取摘要喂给 RAG,不喂原文
- **索引建立时机**:`init` 时一次性建;`/创作` 章节完成后增量更新 `past_chapters` 摘要

### 4.6 Prompt 模板加载

`prompts/<role>.md` 是纯 Markdown,**加载后注入 LangChain `ChatPromptTemplate`**。
查找优先级:**项目级 `.writer/prompts/` > 用户级 `~/.writer/prompts/` > 内置 `prompts/`**。

## 五、LLM Provider(L4 层)

### 5.1 三档模型路由

| 档位 | 默认模型 | 用途 | 触发条件 |
| --- | --- | --- | --- |
| 高端 | `deepseek-v3` 或 `glm-4-plus` | 大纲 / 决战 / 蜕变 / 钩子设计 | Outliner 节点、HookDesigner Skill、关键节(决战段) |
| 中端 | `deepseek-v3` 或 `qwen-plus` | 常规章节正文 / 审核汇总 | Writer 节点、ReviewGate 节点 |
| 轻量 | `glm-4-flash` 或 `qwen-turbo` | 格式整理 / 字数 / 梗概 / 校对 | Proofread 节点、字数统计、梗概改写 |

### 5.2 模型选择策略

- 通过 `WRITER_LLM_HIGH / MID / LOW` 环境变量覆盖
- **缺省配置:全部用 deepseek**(能力 + 价格折中最好)
- 通过 `~/.writer/config.toml` 文件持久化(用户首次选择后记忆)
- 切换 Provider 不改代码,只改 `WRITER_LLM_BASE_URL` + `WRITER_LLM_API_KEY`

### 5.3 L4 接口契约(L3 唯一看到的接口)

```python
class ChatRequest(TypedDict):
    model_tier: Literal["high", "mid", "low"]
    messages: list[Message]
    tools: list[ToolSchema]      # 仅 Writer/Outliner 等需要工具的节点用
    response_format: Literal["text", "json"]
    stream: bool
    max_tokens: int
    temperature: float

class ChatResponse(TypedDict):
    content: str | dict
    usage: UsageInfo
    finish_reason: Literal["stop", "length", "tool_calls"]
```

L3 节点**永远不感知**模型名字——它只说"我要 high 档,流式,要 JSON"。L4 内部决定调用哪个具体模型。

### 5.4 流式输出

- L3 → L2:逐 token 推 `on_token` 回调
- L2 → L1:通过 `Rich` Live 渲染
- **章节正文用流式**(逐字实时);**大纲 / 审核报告用一次性**(避免边写边看)

### 5.5 失败重试

- 5xx / 网络错误:指数退避 3 次(1s / 4s / 16s)
- 4xx / 上下文超限:**降档**(high → mid → low;mid → low)
- 内容截断(finish_reason=length):转人工,报 `LLM_OUTPUT_TRUNCATED`

## 六、错误处理 / 可观测性

### 6.1 错误分类与策略

| 错误类型 | 来源 | 处理 | 用户感知 |
| --- | --- | --- | --- |
| `LLM_TIMEOUT` | L4 | 3 次指数退避;终失败标记节点失败 | "调用 deepseek 超时,3 次重试后失败,稍后重试" |
| `LLM_OUTPUT_TRUNCATED` | L4 finish_reason=length | **不重试**,转人工 | "[!] 输出被截断,可能是 max_tokens 不足或模型溢出" |
| `LLM_CONTEXT_OVERFLOW` | L4 prompt 超限 | 自动降档 + 压缩 RAG 结果 | (内部记录,用户不感知) |
| `CHECKPOINT_CORRUPTED` | L2 | 备份旧 db,新建会话 | "上一会话状态损坏,已备份到 .bak,新开会话" |
| `TOOL_EXEC_FAILED` | L3 工具调用 | 节点级 retry 1 次,失败转人工 | "[!] 工具 Y 失败:X,是否继续?" |
| `STAGE_VALIDATION_FAILED` | L3 review_gate | **回流到上游**(WriterState.retry_count++) | "[i] 第 N 章审核未过,自动回流到 编剧顾问(已重试 X/3 次)" |

### 6.2 重试上限与回滚

- **节点级**:每个节点最多重试 2 次(LLM 错误 + 工具错误)
- **任务级**:整章生成最多重试 3 次(`retry_count` 字段)
- **硬终止**:超过任一上限 → 写 `state["failed"] = True`,L2 把状态回卷到**进入章节前的最后一个 checkpoint**,提示用户介入

### 6.3 日志

- **开发模式**(`WRITER_LOG=debug`):全量 LangGraph + LangChain debug 流到 stderr
- **生产模式**(默认):只有 INFO/WARN/ERROR 写到 `~/.writer/logs/<date>.log`
- **结构化日志**:JSON Lines,字段 `ts / level / layer / node / event / project / chapter`
- **日志轮转**:单日 > 50MB 自动压缩,保留 7 天

### 6.4 可观测性

- **不接入外部 APM**(本地 CLI,无意义)
- L1 输出"任务时间线"面板:每个节点起止时间、token 消耗、模型档位
- 章节末尾自动汇总:`耗时 Xmin / 输入 Y tokens / 输出 Z tokens / 调档次数 / 重试次数`

### 6.5 L1/L2 错误通信

- L3 → L2:沿用 LangGraph 的 `NodeInterrupt` 机制
- L2 → L1:Rich 渲染 `[!]` 红、`[i]` 蓝、`[OK]` 绿
- 用户中断(Ctrl+C):L2 强制 flush 当前 checkpoint,下次 `writer` 自动恢复

## 八、题材分支（2026-07-06 起新增）

题材作为一等公民从 `/init` 进入引擎，[OpenSpec `fea-genre-aware-init`](../openspec/changes/fea-genre-aware-init/proposal.md) 已落地：

- **入口**：`/init <name> --dir <dir> [--genre 历史|言情|玄幻] [--force]`，缺失 `--genre` 时通过 Typer 交互 prompt 提供 `历史 / 言情 / 玄幻 / 其他(自定义)` 四选一
- **持久化**：`AGENT.md` 增加 `题材: <genre>` 行，由 `EngineSession.set_project_root()` 自动重读到 `session.project_genre`
- **派生**：调用方读 `AGENT.md` 的 `题材:` 行后,显式把 `genre` 传给 `production_deps(project_root=..., genre=...)` —— `production_deps` 是纯工厂,自己不再读 `AGENT.md`:
  - `HistoryConsultant` → 历史（chapters 用 `史实:` / `虚构:` 前缀）
  - `XuanhuanConsultant` → 玄幻（chapters 用 `境界<N>` 前缀）
  - `RomanceConsultant` → 言情（chapters 用 `节拍<N>` 前缀）
  - `StoryConsultant` → 兜底四幕（`other` 路径）
