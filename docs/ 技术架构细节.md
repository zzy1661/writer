## 一、设计概要(三大议题决定表)

| 议题 | 决策 | 替代 |
| --- | --- | --- |
| Q3 上下文拼装 | 静态骨架 + 动态 RAG top-up(混合)| 纯静态 / 纯动态 / 角色自适应子图 |
| Q3 滑动窗口 | 全量但按需金字塔(近 3 章 + 中 5-10 章 + 远 10+ 章实体卡)| 仅近 N 章 / 全量展开 |
| Q3 Token 预算 | 按角色分配(Outliner 8k / Writer 30k / Proofread 12k / ReviewGate 16k)| 全局 30k / 200k 优先 |
| Q3 top-up 位置 | 独立 `prep_context` 节点(主节点前)| 主节点内调 / 自适应子图 |
| Q4 工具范围 | 范围 C:文件 IO + 语义 IO + 项目专用 + shell + git | 范围 A/B/D |
| Q4 注册机制 | LangGraph `ToolNode` + `@tool` 装饰 | 自建 Registry / 混合 MCP |
| Q4 权限模型 | cwd 内免 confirm / cwd 外 confirm | 角色白名单 / 全 confirm |
| Q5 质控层数 | 3 道门 + 跨章节审计 | 单门 / 写中+写后 / 卷后自动 |
| Q5 自评维度 | 5 维度评分表(大纲/人物/伏笔/节奏/风格)| 单维度 / 三轮独立角色 |
| Q5 跨章节审计 | 卷/重大节点手动(`/跨审`)| 自动 / 双重触发 |

**贯穿原则**:
- RAG top-up 必须前置且轻量,主节点不感知检索
- 工具按"cwd scope"判定,所有 path 强制 `Path.resolve()` 防穿透
- 写完一章 ≥ self-critique 评分表通过(或上游修订 2 次达到下限),才送到下游

## 二、Q3 记忆 / 上下文管理

### 2.1 上下文拼装流水线

```
prep_context 节点                主节点 (Outliner / Writer / ...)
   │                                  │
   │  收集信号                        │  读取 state.context
   ▼                                  ▼
┌──────────────┐  写入  ┌─────────────────────┐
│ 拼装:        │ ─────► │ state.context        │
│  · 角色任务  │        │  (已有就用,没有就     │
│  · canon 块  │        │   用 state.context_template)│
│  · 上下轮摘要│        └─────────────────────┘
│  · top-up   │
└──────────────┘
        │
        ▼
   角色预算 token 上限检查,
   超预算 → 压缩 RAG 结果,二次裁减
```

### 2.2 prep_context 输入 / 输出

**输入**(`state` 部分字段):
- `chapter_id`、`is_historical`、`stage`
- `state.context_template`(角色任务的固定结构)
- `state.draft`(若存在,取最后 N 段作为续写上文)

**输出**(写入 `state.context`):

```python
class ContextPack(TypedDict):
    system_block: str     # 角色 prompt + 工具 schema + 输出 schema
    canon_block: str      # RAG 检索结果(canon/personas/world)
    history_block: str    # 金字塔摘要
    task_block: str       # 本节点具体任务
    topup_applied: list[str]   # 触发了哪几个 top-up 条件(可观测)
    token_audit: TokenAudit    # 本次各段 token 用量
```

### 2.3 金字塔滑动窗口(具体分级)

| 层级 | 范围 | 形式 | Token 上限 |
| --- | --- | --- | --- |
| **L0 近** | 当前章 + 前后各 1 章 | 章节级摘要(50-200 字/章)+ 关键事件 bullet | 3k |
| **L1 中** | 往前 4-9 章 | 章级摘要更简版(30-100 字)+ 人物/伏笔条目 | 4k |
| **L2 远** | 10 章及之前 | 仅人物履历 + 伏笔状态表 + 信物流转表 | 2k |
| **canon** | 全书 | RAG 检索结果(动态截断到角色预算余量) | 见表 §2.4 |

**生成时机**:
- L0 / L1:`/写` 章节定稿后,**章节完成后** 生成(LLM 跑一次摘要写回 `chapter_summaries.json`)
- L2:跨章节审计时汇总(不常更新;`/跨审` 时一并刷新)

### 2.4 按角色分配 Token 预算

| 节点 | 总预算 | 分配 |
| --- | --- | --- |
| prep_context | — | 内部计算,不计预算 |
| plan_outline | **8k** | system 2k + canon 4k + history 1k + task 1k |
| write_chapter | **30k** | system 2k + canon 4k + history 8k(L0 全)+ task 2k + draft 残段 14k |
| proofread | **12k** | system 1.5k + canon 2k + 待校正文 8k + 校对说明 0.5k |
| history_check | **10k** | system 1.5k + canon(含 history 索引)3k + 待审正文 5k + 史实条目 0.5k |
| review_gate | **16k** | system 2k + 三个 sub-review 报告 10k + 大纲/历史 4k |

**超预算策略**(顺序):
1. 压 RAG(去重 + 截断到 top_k)
2. 压 history(L2 → 标签化,L1 → 删人物次要 trajectory)
3. 压 draft 残段(只保留最近 N 段,N 自适应)
4. 仍超 → 报 `CONTEXT_COMPRESSED`(不重试,继续往下走)

### 2.5 prep_context 节点契约

```python
def prep_context(state: WriterState) -> dict:
    """LangGraph 节点。返回部分 state 更新。"""
    signals = collect_signals(state)        # 拼装信号
    canon = rag.query_canon(...)             # 静态 canon
    history = build_pyramid(state.chapter_id)
    topup = maybe_top_up(signals, canon)     # §2.6 触发条件
    context = pack(
        system_block=load_system(state.stage),
        canon_block=canon,
        history_block=history,
        task_block=load_task(state.stage, state.chapter_id),
        token_budget=ROLE_BUDGET[state.stage],
    )
    return {
        "context": context,
        "topup_applied": [t.name for t in topup],
        "token_audit": context.token_audit,
    }
```

### 2.6 top-up 触发条件

| 条件 | 触发内容 |
| --- | --- |
| `chapter_id` 在「待回收」伏笔列表 | 该伏笔完整条目(描述 + 埋设章节 + 关联人物) |
| `chapter_id` 引入新人物 | 该人物完整档案 |
| `chapter_id` 是卷首章 | 当前卷的总体纲要 |
| `chapter_id` 是上一章结尾引入的悬念 | 上一章最后 200 字原文 |
| `chapter_id` 与重大信物状态有关 | 信物状态卡(谁当前持有) |

触发器由 prep_context 计算,用户不可见;`state.topup_applied` 暴露给 L1 观察。

### 2.7 与 Spec 1 的关系

- Spec 1 §四 4.1 ~ 4.6 仍然有效(索引结构、检索 API、嵌入、Prompt 模板加载)
- 本节新增:
  - **金字塔滑动窗口**(具体三档)
  - **`prep_context` 节点**(在主节点前)
  - **按角色预算**(具体数值)
  - **top-up 触发条件**
  - **超预算压缩策略**

## 三、Q4 工具注册

### 3.1 工具清单(范围 C)

共 14 个工具,分 4 组。

**A. 文件 / 目录(核心 IO)**

| 工具 | 摘要 | 输入 | 输出 |
| --- | --- | --- | --- |
| `read_file` | 读文件 | `path` | `content` |
| `write_file` | 写文件(覆盖) | `path`, `content` | `bytes_written` |
| `append_file` | 追加写入(用于修订记录) | `path`, `content` | `bytes_written` |
| `list_dir` | 列表目录 | `path` | `entries: [name, type, size]` |
| `search_files` | 按 glob/模式查文件 | `pattern` | `paths: list[str]` |
| `mkdir` | 新建目录(agent 创建新子目录用) | `path` | `created` |

**B. Markdown / 字数 / 一致性(语义 IO)**

| 工具 | 摘要 | 输入 | 输出 |
| --- | --- | --- | --- |
| `parse_markdown` | 解析大纲/人物/世界观 | `path`, `query_section?` | 结构化 dict |
| `wordcount` | 字数统计 | `path` | `chars`, `tokens` |
| `consistency_check` | 调用一致性校验器 | `text`, `anchor` | `Conflict[]` |

**C. 项目内部专用(状态管理)**

| 工具 | 摘要 | 输入 | 输出 |
| --- | --- | --- | --- |
| `rag_query` | 读 canon / world / personas | `chapter_id`, `k` | `CanonSlice` |
| `persona_search` | 按名查人物档案 | `name` | `Persona` |
| `foreshadow_update` | 创建/更新伏笔 | `op`(create/recover/discard), `data` | `foreshadow_id` |
| `chapter_register` | 写章节到 `正文草稿/` 或 `正文/` | `chapter_id`, `draft`, `status` | `path` |

**D. 外部操作(默认关闭,需 L1 显式开启)**

| 工具 | 摘要 | 默认 |
| --- | --- | --- |
| `shell_exec` | 执行 shell 命令(仅 cwd 内或允许列表) | 关闭 |
| `git_ops` | git add/commit(默认在工作流内自动 commit) | 开启(push 需 confirm) |

### 3.2 注册机制

```python
from langchain_core.tools import tool

@tool
def read_file(path: str) -> str:
    """读文件内容(path 必须在 cwd 内,或用户已授权路径)"""
    abs_path = Path(path).resolve()
    _ensure_in_scope(abs_path)   # 权限拦截
    return abs_path.read_text(encoding="utf-8")

tool_registry = [
    read_file, write_file, append_file, list_dir,
    search_files, mkdir,
    parse_markdown, wordcount, consistency_check,
    rag_query, persona_search, foreshadow_update,
    chapter_register,
    shell_exec, git_ops,        # 默认按 §3.4 启用
]

tool_node = ToolNode(tool_registry)
```

工具 schema 自动推到 LLM(LLM 直接调 `tool_calls`),**不需要在 prompt 里手写工具列表**。

### 3.3 权限模型(基于 cwd scope)

```
                      调用
                       │
                       ▼
              ┌────────────────────┐
              │  路径解析          │
              │ (resolve + scope)  │
              └────────┬───────────┘
                       │
        ┌──────────────┴───────────────┐
        ▼                              ▼
  在 project_root 内               在 project_root 外
        │                              │
        ▼                              ▼
   无须 confirm                    L1 Rich 提示 confirm
   直接执行                          用户 [y/N] / [y/n]
                                       │
                              ┌────────┴────────┐
                              ▼                 ▼
                            同意              拒绝
                              │                 │
                              ▼                 ▼
                            执行            报 ToolDeniedError,
                                            LangGraph 节点
                                            retry 1 次后转人工
```

**关键规则**:
- `project_root` 在 `init` 时绑定 = `cwd`。会话期不切换。
- 所有 path 在解析后立即与 `project_root` 比较,绝对解析(`Path.resolve()`),防 `..` 穿透
- `shell_exec` 命令先经 sanitization,危险操作 `rm -rf`, `chmod 777`, `mkfs.*` 直接拒绝
- `git_ops` 默认 commit 自动,push 永远 confirm(防误操作)

### 3.4 工具默认状态

| 工具组 | 默认 | 配置入口 |
| --- | --- | --- |
| A. 文件/目录 | **全部开启** | — |
| B. 语义 IO | **全部开启** | — |
| C. 项目专用 | **全部开启** | — |
| `shell_exec` | **关闭** | `WRITER_TOOL_SHELL=1` 开启 |
| `git_ops`(commit) | **开启** | `WRITER_TOOL_GIT=0` 关闭 |
| `git_ops`(push) | **关闭** | `WRITER_TOOL_GIT_PUSH=1` 开启 |

### 3.5 工具结果流转

```
LLM 调 tool_calls
    │
    ▼
LangGraph ToolNode
    │
    ├─→ 写入 state.tool_calls_history(list)
    ├─→ 写入 state.last_tool_result(单条)
    └─→ Rich 实时打印(RUN_STATUS 区)
         │
         ▼
        [OK] read_file: /大纲/大纲.md (4.2k chars)
```

主节点下一轮 LLM 调用**会自动看到**`ToolMessage` 出现在 `messages` 里(LangGraph 自动注入)。

### 3.6 错误处理

| 错误 | 来源 | 处理 |
| --- | --- | --- |
| `TOOL_DENIED` | 权限拦截 | 节点级 retry,LLM 收到"用户拒绝 Y,请换路径/工具"消息 |
| `TOOL_TIMEOUT`(>30s) | 工具执行 | 节点级 retry 1 次,失败标 `[!]` 转人工 |
| `TOOL_NOT_FOUND` | path 不存在 | LLM 收到报错消息,通常应改 path(由 LLM 自纠) |
| `TOOL_OUTPUT_TOO_LARGE`(>50k chars) | LLM 调了读全文 | 自动截断 + 提示"已截断,请用 search_files 分段读" |

### 3.7 与既有文档的关系

- DESIGN.md §四 / Spec 1 §三 都假定 L3 调用"工具",本节把工具集具体化
- DESIGN.md §七 伏笔库自动回收:本节 `foreshadow_update` 工具实现
- DESIGN.md §八 审核:本节 `consistency_check` / `wordcount` 是审核报告的来源

## 四、Q5 长任务质量保证

### 4.1 三道门 + 跨章节审计(总图)

```
                   ┌────────────────────┐
                   │  plan_outline       │  Spec 1
                   └─────────┬──────────┘
                             ▼
   ┌── self-critique 回路(本节新增)───────────────────┐
   │  ┌────────────────────┐    ┌──────────────────┐  │
   │  │ write_chapter (v1) │ ──►│ score(v1, 5 维度) │  │
   │  └────────────────────┘    └────────┬─────────┘  │
   │       ▲                             │            │
   │       └───── 修订(if score 不足) ────┘            │
   └─────────┬──────────────────────────────────────────┘

   ┌─────────────────────────────────────────────────┐

              └────────────────────┘
```

### 4.2 self-critique 评分表(WriterNode 内部)

```python
class DimScore(TypedDict):
    score: int                        # 0-10
    issues: list[str]                 # 修补点
    suggestion: str                   # 一句修订建议

class SelfCritiqueReport(TypedDict):
    pass_overall: bool
    dimensions: dict[str, DimScore]
```

**5 个维度**:

| key | 含义 | 通过阈值 |
| --- | --- | --- |
| `outline_fidelity` | 大纲符合度(本章事件是否对得上) | ≥ 7/10 |
| `character_voice` | 人物语言一致性(口头禅、性格、年龄) | ≥ 7/10 |
| `foreshadow_handling` | 伏笔标记(本章该埋是否埋、该回收是否回收) | ≥ 7/10 |
| `rhythm` | 金三角节奏(开头-中段-钩子) | ≥ 7/10 |
| `style_match` | 风格匹配度(对参考风格的余弦相似度) | ≥ 7/10 |

`pass_overall = sum(d["score"] >= 7 for d in dimensions.values()) >= 4` —— 至少 4 个维度 ≥7 通过(允许 1 维度未达,由下游 review_gate 兜底)。

### 4.3 WriterNode 流程

```
v1 = write(state)                          # 编剧顾问
score = self_critique(v1, state.context)   # 编剧顾问另一路
    ↓
if score["pass_overall"]: accept v1
elif retry < 2 and any_dim < 6:
    v2 = revise(v1, score["suggestions"])  # 编剧顾问修订
    score2 = self_critique(v2, ...)
    accept best of (v1, v2)
else: accept v1 + log issues  # 不卡死,推到 review_gate
```

**成本控制**:最多 2 次修订(写 + 评 + 改 + 评),不上不收敛的重试。

### 4.4 风格参考模板

风格一致性的关键是**抽出参考模板**,而不是靠 prompt 描述风格。

**何时建模板**:
- `init` 时:用户上传 3-5 个参考章节(可选),或指定小说作为参考
- `/风格` 命令:任何时候刷新

**模板字段**(`~/.writer/style/<book-id>.json`):

```python
class StyleTemplate(TypedDict):
    book_id: str               # 参考书名
    source_chapters: list[str] # 引用的章节路径
    avg_sentence_len: float    # 平均句长
    para_length: float         # 段落均值
    pov_distribution: dict     # 第一/三人称占比、视角切换频率
    vocab_level: str           # 文言/半文半白/现代白话
    dialogue_ratio: float      # 对话占比
    unique_words: int          # 词汇丰富度估计
    embedding_centroid: list[float]   # 风格向量(嵌入空间中心)
```

**运行时**:
- `style_match` 维度 = 新写章节向量与 `style_template.embedding_centroid` 的余弦相似度
- 风格嵌入复用 `bge-large-zh`(与 RAG 嵌入共用同一个模型)

### 4.5 跨章节审计(`/跨审`)

**触发方式**:
- 用户在 REPL 输入 `/跨审`(手动)
- 可选参数:`/跨审 1.1-1.10`(指定章节范围)
- 不在常规流程里——只在"卷结尾 / 重大决战前"等关键时刻由用户主动调用

**审计维度**:

| 维度 | 检查内容 | 工具 |
| --- | --- | --- |
| 人物弧光 | 主角/关键配角自首章到本章的态度、能力、关系演变 | LLM 对每章摘要做时序分析 |
| 伏笔收支 | 范围段内每个伏笔 ID:是否按计划埋设 / 触发 / 回收 | LLM 读 `伏笔/伏笔库.md` |
| 信物流转 | 信物当前持有人与上次记录的对比 | `consistency_check`(Q4) |
| 时代线 | 事件时间线是否矛盾(年龄、季节、纪年) | `consistency_check` + LLM |
| 主题违和 | 局部主题(亲情/复仇/成长)是否偏离大纲锚定 | LLM 读大纲对比 |

**输出**(`大纲/大纲.md` 顶部「修订记录」追加):

```markdown
- 2026-07-XX-HH:MM 跨审(章节范围:1.1-1.10):
  - 人物弧光: 主角态度变化符合大纲(评分 8/10),配角 A 偏离(评分 4/10)
  - 伏笔:F003 按计划埋设 ✓,F005 触发但未回收 → 建议 1.11 处理
  - 信物: 玉簪(ch1.4)→ 主角(ch1.8)→ 配角 A(ch1.9), 持主线贯穿 OK
  - 时代线: 无矛盾
  - 主题违和: 无
```

**成本**:一次性调用(可能分多次,跨 5-10 章通常 2-3 次),不依赖存储中间结果。
