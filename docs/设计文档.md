# Writer Agent 设计文档

> 本文档为系统设计,**不包含实现细节**。所有架构、命令、技能、子代理定义以此为准。
> 后续若有调整,以本文件为准;实现期间发现冲突需回头修订本文档。

---

## 一、系统认定的项目目录

`init` 命令生成的项目根目录下,包含以下目录(部分可选):

| 目录 | 必选 | 用途 | 主要文件 |
| ---- | ---- | ---- | -------- |
| `创意/` | ✓ | 用户零散灵感、点子、未成形的素材 | 多个 `*.md`,按主题/时间组织 |
| `大纲/` | ✓ | **唯一正典大纲**,所有写作以此为准 | `大纲.md` |
| `目录/` | ✓ | 卷-章两级目录,**不含节** | `目录.md` |
| `梗概/` | ✓ | 故事梗概 | `五十字梗概.md`(~50 字)<br>`五百字梗概.md`(~500 字)<br>`章级梗概.md`(每章 1-3 句) |
| `人物/` | ✓ | 人物档案 | `人物档案.md`(或 `人物/<姓名>.md` 分文件) |
| `世界观/` | ✓ | 时代背景、地理、势力、设定 | 按子主题拆分:`时代背景.md`、`地理.md`、`势力.md`、`规则.md` 等 |
| `伏笔/` | ✓ | 伏笔库 | `伏笔库.md` |
| `正文草稿/` | ✓ | 写作中的草稿 | `卷N_第M章_标题.md` |
| `正文/` | ✓ | 已定稿章节(用户审核通过后从草稿移入) | `卷N_第M章_标题.md` |
| `史实/` | ✗ | **历史题材专用** | `年表.md`、`人物.md`、`事件.md`、`考证.md` |
| `AGENT.md` | ✓ | 项目核心状态总览(非目录,根文件) | 单文件 |

### 1.1 目录命名硬约束

- 全部使用中文目录名,**不加空格、不用下划线**
- 文件名采用下划线分隔卷章:`卷N_第M章_标题.md`
- 修正意见不单独成目录;写作期反馈写入 `大纲/大纲.md` 顶部「修订记录」

## 二、AGENT.md 规范

`AGENT.md` 是项目的**唯一状态总览**,放在项目根目录。它是 Agent 启动时最先读取的文件。

### 2.1 文件结构

```markdown
# 《书名》Agent 状态

> 本文件由 Agent 维护,记录项目核心信息。
> 不要手工删除;改核心创意请用 `init` 重置或 `大纲` 命令修订。

## 项目元数据

- 书名:
- 创建日期:
- 题材: <类型> / <子类型>(如 武侠 / 五代十国背景)
- 时代: <朝代> / <起止年份>
- 字数目标: <万字>
- 当前进度: <已写字数> / <总字数> 万字(<百分比>%)
- 当前章节: 卷N 第M章 <标题>
- 当前状态: <创意 | 大纲 | 目录 | 写作 | 修订>

## 核心创意

<用户最初提供的核心创意原文,逐字保留>

## 50 字梗概

<一句话,约 50 字>

## 500 字梗概

<一段话,约 500 字>

## 主要人物

- 主角: <姓名>(<年龄>,<一句话定位>)
- 对手: ...
- 关键配角: ...

## 题材定位

- 类型: <古言/现言/悬疑/科幻/武侠/无限流/耽美/现实向/...>
- 题材规范: <由 GenreLocator 输出>
- 受众偏好: ...
- 避雷设定: <不可触碰的红线>

## 决策记录

- YYYY-MM-DD-HH:MM: <决策内容>
```

### 2.2 维护规则

- `AGENT.md` 由 Agent 自动维护,**用户不应手动编辑**
- 每次 `init`、`大纲`、`目录`、写完一章后,Agent 同步更新对应字段
- 历史版本以 git 管理(用户自行决定是否初始化仓库)

## 三、Init 流程

### 3.1 触发

启动 `writer` 进入 REPL 后,用户输入 `init` 命令。

### 3.2 步骤

```
1. 检查 cwd 是否已存在 AGENT.md
   ├─ 存在:提示「项目已初始化,使用 reset 命令重置」→ 退出
   └─ 不存在:继续

2. 询问书名(可跳过,默认「未命名」)

3. 询问核心创意 ★必填
   用户输入自由文本,可多行。
   Agent 不打断,等用户输入 `/done` 结束。

4. 调用 GenreLocator Skill
   ├─ 自动判断题材
   ├─ 若是「历史」/「架空历史」/「半架空历史」等历史相关题材:
   │    └─ 创建 史实/ 目录及其子文件模板
   └─ 否则:跳过 史实/ 目录

5. 调用 LLM 生成
   ├─ 50 字梗概
   ├─ 500 字梗概
   ├─ 主要人物候选(主角/对手/2-3 配角)
   └─ 题材规范、受众偏好、避雷设定

6. 创建所有目录骨架
   ├─ 创意/ 大纲/ 目录/ 梗概/ 人物/ 世界观/ 伏笔/ 正文草稿/ 正文/
   └─ (历史类) 史实/

7. 写入 AGENT.md(填充以上字段)

8. 提示完成
   > 已初始化项目《xxx》,可输入 `大纲` 开始下一步。
```

### 3.3 Init 失败的回滚

任何步骤失败 → 删除已创建的部分目录,保留 cwd 不变。

## 四、命令清单(Command)

命令是用户**主动调用**的入口,以 `/` 开头。

### 4.1 分阶段 MVP 命令

| 阶段 | 命令 | 别名 | 输入 | 行为 |
| ---- | ---- | ---- | ---- | ---- |
| 阶段一 | `/帮助` | `/help` | (无) | 显示基础命令 |
| 阶段一 | `/退出` | `/exit` `/quit` `/q` | (无) | 退出 REPL |
| 阶段一 | `/init` | - | 故事梗概 + 题材选择 | 生成 `AGENT.md`、项目目录和初版故事大纲 |
| 阶段一 | `/目录` | `/toc` | (无) | 基于大纲生成全文分卷目录 |
| 阶段二 | `/创意` | `/idea` | 创意线索 | 追加到创意库 |
| 阶段二 | `/伏笔` | `/foreshadow` | 伏笔描述 | 追加到伏笔库 |
| 阶段二 | `/大纲` | `/outline` | (无) | 基于创意库、伏笔库生成或重生成故事大纲 |
| 阶段二 | `/骨架` | - | (无) | 基于大纲和目录生成每章开头、结尾 |
| 阶段三 | `/创作` | `/write` | 卷/章定位 | 写作某一卷或章节;默认当前卷,写完一卷后停下审核 |
| 阶段三 | `/审核` | `/review` | `大纲`/`骨架`/`正文` | 分别审核大纲、骨架、正文,并驱动正文自动修复 |

### 4.2 后续扩展命令

| 命令 | 别名 | 行为 |
| ---- | ---- | ---- |
| `/状态` | `/status` | 显示 AGENT.md 摘要 |
| `/史实` | `/history` | (仅历史题材)查询史实,由历史顾问回答 |
| `/钩子` | `/hook` | 调用 HookDesigner Skill,设计前三章开篇钩子 |
| `/分镜` | `/storyboard` | 调用 PlotVisualizer,生成文字分镜 |
| `/人设图` | `/avatar` | 调用 CharacterImagePromptGen,生成角色绘图 prompt |

### 4.3 命令实现规则

- 所有命令都**先校验项目状态**(`AGENT.md` 存在性)
- 命令执行的中间过程**实时打印**(用 Rich Live 渲染流式 LLM 输出)
- 命令完成后**写入 AGENT.md 对应字段** + 在 `大纲/大纲.md` 顶部追加修订记录

## 五、技能清单(Skill)

技能是 Agent 内部调用的能力,**用户不直接触发**(除非通过 `/钩子` 等辅助命令)。

### 5.1 企划设定类 Skill

#### Skill-01: GenreLocator(题材定位)

- **触发**:`/init` 时自动调用
- **输入**:核心创意文本
- **输出**:题材类型、子类型、题材规范、受众偏好、避雷红设定
- **写入**:`AGENT.md` 题材定位段落

### 5.2 商业化网文配套 Skill

#### Skill-02: HookDesigner(开篇钩子设计)

- **触发**:`/创作` 写前三章时自动调用
- **输入**:核心创意 + 50 字梗概 + 第一章标题
- **输出**:前三章每章开篇 200-500 字的钩子设计(悬念/情绪冲突/反差)
- **写入**:`大纲/大纲.md` 的「钩子设计」子节

#### Skill-03: CliffhangerBuilder(章节末尾留钩子)

- **触发**:`/创作` 每章末尾自动调用
- **输入**:已写章节内容摘要 + 下一章大纲
- **输出**:本章末 100-300 字的钩子(可以是悬念句、未完动作、伏笔揭示)
- **不直接写入正文**,而是作为子代理生成正文的最后一段指引

#### Skill-04: HotSpotIntegrator(热点桥段融合)

- **触发**:由编剧顾问在写正文时按需调用
- **输入**:当前章节梗概 + 题材类型
- **输出**:1-3 个可融合的热门桥段建议(必须不与主线冲突)
- **不强制**,由编剧顾问判断采纳

#### Skill-05: CommentSimulator(读者反馈模拟)

- **触发**:`/审核` 时自动调用
- **输入**:章节正文 + 章节梗概
- **输出**:模拟 3-5 种读者视角的反馈(追订读者、弃书读者、评论区吐槽点)
- **写入**:审核报告的「读者视角」部分

### 5.3 跨工具联动扩展 Skill

#### Skill-06: PromptRefiner(提示词自动优化)

- **触发**:子代理每次 LLM 调用前**自动**调用
- **输入**:原始 prompt + 上一轮生成结果(若存在)
- **输出**:优化后的 prompt
- **设计**:这是一个**后台循环**,首先生成 → 评估 → 优化 prompt → 重生成,直到满意或达上限

#### Skill-07: PlotVisualizer(剧情转文字分镜)

- **触发**:`/分镜` 命令
- **输入**:章节正文
- **输出**:文字分镜脚本(场景、人物、动作、镜头描述),适配 AI 视频生成工具

#### Skill-08: CharacterImagePromptGen(人设转绘图 prompt)

- **触发**:`/人设图` 命令
- **输入**:人物档案
- **输出**:Stable Diffusion / Midjourney 兼容的角色绘图 prompt

#### Skill-09: MultiChapterBatchWriter(批量多章节生成)

- **状态**:已移出 MVP 用户命令集,保留为未来内部能力候选
- **输入**:章节范围(如 `1.1-1.5`)
- **输出**:多章节正文,每章独立走「创作→审→改」流程
- **约束**:每章写完都过审核,不通过的章节标红,告知用户

## 六、子代理清单(Sub-Agent)

子代理是**专业角色**,有独立的 prompt、专属 Skill 调用权、独立上下文。命令和技能在执行时**委派**给一个或多个子代理。

### 6.1 编剧顾问(StoryConsultant)

- **职责**:剧情设定、情节创造、对话撰写、场景设计
- **核心原则**:
  - **有想象力** — 主动提出情节建议、人物冲突
  - **遵循大纲** — 所有创造不超出 `大纲/大纲.md` 范围
  - **遵循故事框架** — 50 字 / 500 字梗概是创作的最高约束
- **调用 Skill**:HookDesigner、CliffhangerBuilder、HotSpotIntegrator、PromptRefiner
- **被触发**:
  - `/大纲`、`/目录`、`/创作`、`/创意`
  - HookDesigner 自动调用时

### 6.2 历史顾问(HistoryConsultant) ★ 历史题材专属

- **职责**:
  - 结合 `史实/` 资料对项目中内容做**史实评判**
  - 回答用户的历史知识查询(`/史实` 命令)
  - 在大纲/正文涉及历史人物、事件、年号、官职时,标注**史实偏离**与**可虚构边界**
- **核心原则**:
  - **史实优先** — 历史人物的命运、关键事件年份不可篡改
  - **虚构可标记** — 允许偏离时给出标注「此处为虚构,史实为...」
  - **可输出新史实材料** — 用户查询时主动整理相关史料
- **调用 Skill**:PromptRefiner
- **被触发**:
  - `/审核` 时自动调用(审核报告必须包含史实检查)
  - `/创作` 时若涉及历史人物/事件,自动调用
  - `/史实` 命令
- **非历史题材**:此子代理**不加载**,减少 token 浪费

### 6.3 校对(Proofreader)

- **职责**:
  - 错别字检查(同音字、形近字)
  - 语句不通顺(主谓、搭配、成分残缺)
  - 标点符号、段落格式
  - 敏感词审查(可选,默认开启)
- **核心原则**:
  - **只改不改意** — 不修改情节、人物、对话内容
  - **保留风格** — 不抹平作者的修辞特色
  - **批量标注** — 用 diff 形式输出,不改写
- **调用 Skill**:无
- **被触发**:
  - `/审核` 时调用
  - `/创作` 完成后自动调用(标点与错别字自动修复,语病标建议)

### 6.4 子代理的协作关系

```
         /创作 写作章节
             │
             ▼
    ┌─────────────────┐
    │   编剧顾问       │ ←── 触发 HookDesigner / CliffhangerBuilder
    │ (StoryConsultant)│ ←── 触发 HotSpotIntegrator(可选)
    └────────┬────────┘
             │ 生成章节草稿
             ▼
    ┌─────────────────┐         ┌─────────────────┐
    │   校对           │ ──────► │   历史顾问       │ (仅历史题材)
    │ (Proofreader)   │         │ (HistoryConsultant)
    └────────┬────────┘         └────────┬────────┘
             │ 校对报告                 │ 史实评判
             ▼                          ▼
         ┌─────────────────────────────────────┐
         │         /审核 综合报告              │
         └─────────────────────────────────────┘
```

- **编剧顾问是主执行者**,其他两个是审核角色
- 三个子代理**上下文隔离**,各自只看自己需要的文档
- 编剧顾问**不感知**校对和历史顾问的存在;它们在审核阶段介入

## 七、伏笔管理

伏笔是独立模块,有完整的生命周期。

### 7.1 伏笔库结构

`伏笔/伏笔库.md` 格式:

```markdown
# 伏笔库

> 所有伏笔的中央登记簿。每个伏笔有唯一 ID、状态、出现章节、回收章节。

## 伏笔列表

### F001 — <伏笔标题>
- 创建日期:YYYY-MM-DD
- 状态: 潜伏 | 触发 | 已回收 | 弃用
- 埋设章节: 卷N 第M章
- 回收章节: 卷X 第Y章
- 描述: <伏笔内容,一句话>
- 关联人物: <姓名1>, <姓名2>
- 关联物品: <物品名>(如有)
- 备注: <回收时的具体方式>

### F002 — ...
```

### 7.2 `/伏笔` 命令

- **输入**:用户描述一段伏笔
- **行为**:
  1. LLM 抽取:伏笔 ID(自增)、标题、描述、关联人物/物品
  2. 状态默认 `潜伏`
  3. 写入 `伏笔/伏笔库.md`
  4. 返回伏笔 ID

### 7.3 自动关联

- `/创作` 完成后,校对子代理**不感知**伏笔
- 编剧顾问写章节时,系统从 `伏笔/伏笔库.md` 抽取**当前章节相关的潜伏伏笔**注入上下文
- `/审核` 时,审核报告检查伏笔回收是否按计划完成

## 八、审核(/审核)流程

### 8.1 审核范围

默认审核 `正文草稿/` 全部内容;可选指定章节(`/审核 1.3`)。

### 8.2 审核维度

| 维度 | 责任子代理 | 输出 |
| ---- | ---------- | ---- |
| 字数达标(每章 ≥ 3000) | 校对 | 字数报告 |
| 错别字 / 语病 | 校对 | 校对报告(diff) |
| 人物一致性(年龄、性格、口头禅) | 编剧顾问 | 一致性报告 |
| 大纲符合度 | 编剧顾问 | 大纲偏离点 |
| 历史准确性 | 历史顾问(仅历史题材) | 史实评判 |
| 伏笔按计划 | 编剧顾问 | 伏笔偏离点 |
| 读者视角 | CommentSimulator Skill | 模拟反馈 |

### 8.3 审核报告输出

写入 `大纲/大纲.md` 顶部「修订记录」,格式:

```markdown
- YYYY-MM-DD-HH:MM 审核(编剧顾问 + 校对 + 历史顾问):
  - 卷一第1章:字数 3200 ✓
  - 卷一第2章:字数 2800 ✗ 需补写
  - 卷一第3章:主角年龄 12 岁与大纲「13 岁」不一致 → 修正建议
```

不通过审核的章节**标红**,提示用户处理。

## 九、AGENT.md 与各目录的同步规则

### 9.1 双向同步

| 事件 | 同步目标 |
| ---- | -------- |
| `init` 完成 | AGENT.md 写入元数据 + 核心创意 + 梗概 |
| `大纲` 生成 | AGENT.md 更新「当前状态」,大纲.md 写入 |
| `目录` 生成 | AGENT.md 更新「当前章节」为第一卷第一章,目录.md 写入 |
| `/创作` 写完一章 | AGENT.md 更新「当前章节」「当前进度」「已写字数」 |
| `伏笔` 创建 | AGENT.md 不动,但伏笔 ID 写入大纲/大纲.md |
| `审核` 完成 | 审核记录写入大纲/大纲.md 顶部,AGENT.md 更新「当前状态」 |

### 9.2 写入约束

- `AGENT.md` 由 Agent 写,**用户只读**
- 任何修改核心创意的操作都先备份 `AGENT.bak.<timestamp>.md`
- `大纲/大纲.md` 是唯一正典,所有写作必须以其为基准
