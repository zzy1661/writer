# 正典文件与多源写入一致性

## 业务背景

项目要求 `大纲/大纲.md` 是唯一正典,`AGENT.md` 是唯一状态总览。人物、世界观、伏笔、史实、正文草稿都会影响后续写作,但不能各自形成互相冲突的事实来源。

## 技术难点

多个命令和 Agent 节点都会写文件: `/init` 写 `AGENT.md`, `/大纲` 写正典大纲, `/伏笔` 写伏笔库, `/创作` 写正文草稿, `/审核` 追加修订记录。若缺少统一写入协议,容易出现以下问题:

- `AGENT.md` 与目录状态不一致。
- 伏笔库已更新,但大纲没有同步记录。
- 审核结果覆盖用户认可的正典内容。
- 中途失败留下半写文件。

## 解决方案

引入项目写入服务 `ProjectRepository`,所有文件操作都通过该服务执行。写入分为三类:

- 正典写入:大纲、目录、人物、世界观、伏笔 ledger。
- 状态写入:`AGENT.md`。
- 工作产物写入:正文草稿、审核报告、章节摘要。

关键写入采用临时文件 + 原子替换,追加型写入保留时间戳和来源。破坏性写入先备份,例如重写大纲前创建 `大纲/大纲.bak.<timestamp>.md`。

> **2026-07-08 实测**:`ProjectRepository` 这个具体类**当前没有实装**(2026-07-09 grep `src/` 找不到),文件 IO 由 builtin Tool(`safe_read_file` / `safe_write_file` / `safe_edit_file`)+ LLM 工具循环的 SKILL.md directive 教学替代。`AGENT.md` 的"结构化读写器"实装在 `src/writer/project/state.py::render_agent_file` / `read_genre_from_agent` / `append_agent_requirements` 三个函数;`大纲/大纲.md` 的"修订记录追加"实装由 LLM 在 SKILL.md directive 教学下做 `safe_edit_file` 替换实现。`append_revision_record` 这种专用 Tool **不在 builtin Tool 清单里**——见 [备忘 13](./13-核心Tool设计.md)。

## 最小 demo / 伪代码

```python
from pathlib import Path
from tempfile import NamedTemporaryFile


class ProjectRepository:
    def __init__(self, project_root: Path) -> None:
        self.project_root = project_root.resolve()

    def resolve(self, path: str) -> Path:
        target = (self.project_root / path).resolve()
        if self.project_root not in [target, *target.parents]:
            raise PermissionError(f"禁止写出项目目录: {target}")
        return target

    def atomic_write(self, path: str, content: str) -> None:
        target = self.resolve(path)
        target.parent.mkdir(parents=True, exist_ok=True)
        with NamedTemporaryFile("w", encoding="utf-8", dir=target.parent, delete=False) as tmp:
            tmp.write(content)
            tmp_path = Path(tmp.name)
        tmp_path.replace(target)

    def append_revision_record(self, message: str) -> None:
        outline = self.resolve("大纲/大纲.md")
        old = outline.read_text(encoding="utf-8")
        marker = "## 修订记录\n"
        new = old.replace(marker, f"{marker}\n- {message}\n", 1)
        self.atomic_write("大纲/大纲.md", new)
```

## 核心依赖版最小代码

```python
from datetime import datetime
from pathlib import Path

from pydantic import BaseModel, Field
import tomli_w


class AgentState(BaseModel):
    book_title: str
    status: str
    current_chapter: str
    decisions: list[str] = Field(default_factory=list)


def update_agent_state(project_root: Path, state: AgentState) -> None:
    # 实现时可渲染成 AGENT.md;这里用 tomli-w 展示结构化写入思路。
    state_path = project_root / ".writer" / "state.toml"
    state_path.parent.mkdir(parents=True, exist_ok=True)
    state.decisions.append(f"{datetime.now():%Y-%m-%d %H:%M} 状态更新为 {state.status}")
    state_path.write_text(tomli_w.dumps(state.model_dump()), encoding="utf-8")


def render_agent_md(state: AgentState) -> str:
    return f"""# 《{state.book_title}》Agent 状态

## 项目元数据

- 当前状态: {state.status}
- 当前章节: {state.current_chapter}

## 决策记录

{chr(10).join(f"- {item}" for item in state.decisions)}
"""
```

## 落地建议

- 文件写入统一使用 `Path.resolve()` 并限制在 `project_root` 内。
- 为 `AGENT.md` 建立结构化读写器,不要依赖脆弱字符串替换。
- 对 `大纲/大纲.md` 顶部「修订记录」提供专用 `append_revision_record()`。
- 命令处理器不直接调用 `path.write_text()`。

## 验收标准

- 任意失败不会留下空文件或半截正典文件。
- 同一次命令产生的多个文件更新要么都成功,要么回滚可恢复。
- `AGENT.md` 可在启动时准确恢复项目状态(`detect_state` + `read_genre_from_agent`)。
- 用户手动文件改动不会被无提示覆盖(LLM 工具循环写文件前先 `safe_read_file`,改 `大纲/大纲.md` 前先 append 修订记录)。

## 2026-07-08 修订:`AGENT.md` 实际格式

旧设计文档假设的 `AGENT.md` 是 Markdown 自由文本,但**当前实装**采用更紧凑的字段化格式(per `src/writer/project/state.py::render_agent_file`):

```markdown
# 长安程序员

Writer Agent 项目状态文件。

## 当前状态

- state: S1
- label: 初始化
- 题材: 历史

## 目录约定

- outline/: 大纲、目录与分卷规划
- manuscript/: 正文草稿
- characters/: 人物设定
- world/: 世界观设定
- notes/: 写作笔记
- 创意/: 故事创意与核心设定
```

**关键字段**:

- `## 当前状态` 块下的 `- state: Sx` 是机器读字段(`detect_state` 优先用文件存在性判断,这里只是参考)
- `- 题材: <genre>` 在 `genre != "other"` 时由 `render_agent_file` 渲染;`read_genre_from_agent` 用正则解析回字符串(`/^题材:\s*(.+)$/`),供 `Engine.refresh_project_genre()` 用
- `## 基本要求` 是 `/init <故事梗概>` 跑 `apply_init_brief` 后由 `append_agent_requirements` 追加

业务字段(`50 字梗概` / `500 字梗概` / `主要人物` / `题材定位` / `决策记录` 等)不再强制写到 `AGENT.md`,改写到 `outline/premise.md` / `outline/volume-plan.md` / `characters/main.md` 等独立文件,LLM 工具循环按需读取。
