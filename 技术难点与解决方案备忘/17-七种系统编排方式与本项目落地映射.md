# 七种系统编排方式与本项目落地映射

## 问题

Agent 系统中常见的七种编排模式(Prompt Chaining / Routing / Parallelization / Evaluator-Optimizer / LangGraph 图状态机 / HITL / Supervisor-Worker)在本项目中分别落地在哪里?如何避免“一种模式走到底”导致的复杂度爆炸或能力不足?

## 业务背景

上一份文档已经给出整体架构选择:`IntentRouter` 前台路由(`RuleBasedIntentRouter` 当前 MVP) + LangGraph 后台 + 业务 Tool + RAG。本文档将该方案再往下沉一层,逐一对应到这七种编排模式,标明它们具体负责哪一段业务、取代了什么朴素做法、带来什么权衡。

七种模式之间不是互斥的,而是按子任务粒度组合使用的。下面给出本项目内的具体落点。

## 技术难点

- 不同编排模式适合不同任务粒度,使用错位会带来过度设计或能力缺位。例如,长章节写作如果只用 Prompt Chaining,会失去条件回流能力;高频命令如果套上完整 LangGraph,反而引入不必要的开销。
- 真实项目中一个工作流往往跨越多种编排模式。如果不在代码层面把模式拆开,后续维护和调试会非常困难。

## 总览:模式-落点-理由

| 编排模式 | 本项目落点 | 主要价值 |
| --- | --- | --- |
| Prompt Chaining | `/init` 元信息沉淀、大纲 → 章纲 → 章纲细化 | 把可拆解的线性流程固化,便于复现与单点缓存 |
| Routing | 前台 `IntentRouter`(MVP=`RuleBasedIntentRouter`)、命令级状态机路由 | 把意图/状态分派到合适的工作流,不堆积复杂度 |
| Parallelization | 多顾问并行审核、多卷章纲并行生成、跨章节审计并发 | 缩短长流程耗时,避免串行依赖 |
| Evaluator-Optimizer | 写作-审核-回流的章节润色循环、伏笔触发校验 | 满足长任务的稳定质量,而非一次性草稿 |
| LangGraph 图状态机 | 章节写作主图、跨章节一致性扫描图、批量写作图 | 显式表达节点、条件边、回流边、checkpoint |
| Human-in-the-Loop | 题材选择、大纲确认、敏感决策、回流确认 | 用户主观偏好与项目长生命周期需要人介入 |
| Supervisor-Worker | 批量章节写作组长、跨章节一致性扫描、写作助手套件集成 | 1 个控制器 + 多个 worker 各负责一段,适合任务分解 |

下文逐一展开。

## 1. Prompt Chaining(链式流水线)

### 落点

- `/init` 元信息沉淀:`题材定位 → 受众推断 → 主题主张 → 写作风格 → 元信息汇总`。每个 LLM 输出当作下一阶段的输入,中间用确定性代码做归一化。
- 章纲生成:`大纲 → 章节标题候选 → 摘要候选 → 排序后写入 章纲`。典型的小型流水线。

### 为什么不用单一长 prompt

- 单一 prompt 完成多步任务时,模型容易跳步、把后续任务混入当前任务。
- 链式可以让每一步单独缓存、独立重跑、按用户反馈局部重做。
- 可以针对每步挑选不同模型和不同 temperature。

### 最小 demo / 伪代码

```python
from dataclasses import dataclass


@dataclass
class InitStep:
    name: str
    prompt: str


INIT_PIPELINE = [
    InitStep("genre",       "根据创意:{idea} 给出题材标签。"),
    InitStep("audience",    "根据题材:{genre} 给出目标读者画像。"),
    InitStep("style",       "根据读者:{audience} 给出写作风格建议。"),
    InitStep("summary",     "汇总:{genre}/{audience}/{style} 生成正典用元信息。"),
]


def chain_init(llm_call, idea: str) -> dict:
    state: dict = {"idea": idea}
    for step in INIT_PIPELINE:
        state[step.name] = llm_call(step.prompt.format(**state))
    return state
```

## 2. Routing(路由分发)

### 落点

- 前台 `IntentRouter`(已在 doc 15):用户输入 → `run_command` / `call_tool` / `start_workflow` / `ask_user` / `answer_directly`。协议见 `writer.routing.IntentRouter`,当前 MVP 实现见 `RuleBasedIntentRouter`。
- 命令级状态机(已在 doc 01):`validate_command_available()` 根据 S0-S5 决定哪些命令可用。
- LangGraph 工作流中的 `route_after_proofread`:`is_historical` 决定是否走 `history_check`。

### 设计要点

- 路由选择越早越好,越前越省 token。
- 路由判定的可观测性要强,记录“为什么走到这条路径”,便于回溯。

### 最小 demo / 伪代码

```python
from typing import Literal
from pydantic import BaseModel


Action = Literal["start_workflow", "call_tool", "ask_user", "answer_directly"]


class RouterDecision(BaseModel):
    action: Action
    target: str | None = None  # 工作流或工具名


def llm_router(llm, user_input: str, project_state: str) -> RouterDecision:
    chain = ROUTER_PROMPT | llm.with_structured_output(RouterDecision)
    return chain.invoke({"input": user_input, "state": project_state})


def dispatch(decision: RouterDecision, writer_graph, tool_registry) -> object:
    if decision.action == "start_workflow":
        return writer_graph.invoke({"workflow": decision.target})
    if decision.action == "call_tool":
        return tool_registry.call(decision.target)
    if decision.action == "ask_user":
        return {"interrupt": True, "question": decision.target}
    return decision.target  # answer_directly
```

### Routing 的终止面:ActionType ↔ Done.reason

上面的 demo 只解决了"router 选 action",但 engine 还要告诉 consumer 这一轮为什么结束。两套 vocabulary 必须显式对齐,否则 REPL / Web UI 渲染会拿不到稳定 vocabulary。

备忘 15 / 16 / engine 实测共同确认:

| ActionType (router 选) | engine 实测默认 Done.reason | 备注 |
|---|---|---|
| `answer_directly` | `answered` | 直接回答,本 turn 结束 |
| `run_command` | `command_pending` | 框架命令已识别,执行交给 CLI stub 或未来 REPL handler |
| `call_tool` | `tool_pending` | 工具已选定,执行留给 ToolNode (等 per 13 接 wire) |
| `start_workflow` | `workflow_pending` | 工作流已选定,启动留给 LangGraph (等 per 04) |
| `ask_user` | `ask_user` | MVP 直接终结,见下方"ask_user 双重身份" |

Done.reason 还有第 6 个值 `aborted`,留给 cancel / Ctrl+C / supervisor 强制中断使用,不由 router 选。

#### ask_user 的双重身份

`ask_user` 同时出现在 ActionType 和 Done.reason 里,但语义不同:

- **ActionType="ask_user"**: router 主动选择"信息不足,反问用户"。此时 engine 发出 `Done(reason="ask_user", payload={"prompt": ...})`,**本 turn 终结**,用户的回答是下一次 turn 的新 `user_input`。
- **等价的真实 Interrupt 路径**: per 14 的 `AgentInterrupt(type="choice"/"text"/"confirm")` 是另一种"问用户",但语义是 **本 turn 暂停 + 同 session 内 resume**(由 LangGraph checkpoint 配套)。

两条路径并存: MVP 用前者(简单),等 per 04 + 14 接好 LangGraph resume 后再补后者(支持复杂选择)。Engine 的 `Event` 层级(`Interrupt` vs `Done(reason="ask_user")`)就是这两条路径的协议层。

#### Engine Event 类型层级作为 Web UI 协议

Engine 已经稳定的 Event 类型(见 `writer.engine.events`):

```
Event
├── TextChunk          # 增量文本
├── ActionEvent        # 路由器选了哪个 ActionType
├── ToolCall / ToolResult  # 工具调用 + 结果(等 per 13 wire)
├── Interrupt          # 等 per 14, resume 型
├── Done(reason=…)     # 终结型(reason 见上表)
└── ErrorEvent         # 异常
```

该层级就是 Web UI 将来要实现的渲染协议——payload 形状稳定,UI 端按 Event 子类型分发。

## 3. Parallelization(并行执行)

### 落点

- **多顾问并行审核**:章节定稿时,校对 + 历史校验 + 文风评估同时跑,最后汇总。
- **多卷并行生成章纲**:获得大纲后,各卷独立并行生成 `目录/卷N.md`,再合并回 `目录/目录.md`。
- **跨章节一致性扫描**:对每章独立跑一致性报告,然后聚合冲突点。
- **RAG 多查询融合**:同一章任务并发检索伏笔 / 人物 / 史实 / 前情摘要,延迟最低路径胜出。

### 设计要点

- 并行任务必须彼此独立,不能写到同一个草稿或同一段 state 字段。
- 聚合阶段要用确定性合并逻辑,不要把多份报告拼接给同一个 LLM,否则会被幻觉带偏。

### 最小 demo / 伪代码

```python
import asyncio
from dataclasses import dataclass


@dataclass
class CheckReport:
    chapter_id: str
    source: str   # "proofread" / "history" / "style"
    issues: list[str]


async def run_one_check(chapter_id: str, source: str) -> CheckReport:
    # 实际中调用不同角色 prompt + 共享上下文切片
    return CheckReport(chapter_id=chapter_id, source=source, issues=[])


async def parallel_review(chapter_id: str, flags: dict) -> list[CheckReport]:
    tasks = [run_one_check(chapter_id, "proofread")]
    if flags.get("is_historical"):
        tasks.append(run_one_check(chapter_id, "history"))
    if flags.get("needs_style"):
        tasks.append(run_one_check(chapter_id, "style"))
    return await asyncio.gather(*tasks)
```

## 4. Evaluator-Optimizer(评估-优化)

### 落点

- **章节写作-审核-回流**:写过草稿进入 `proofread`,失败就回流到 `write_chapter`(已在 doc 04)。这是最经典的 Evaluator-Optimizer 循环。
- **伏笔质量评估**:章节定稿前由评估者 LLM 检查本章伏笔:该埋则埋、该挖则挖,缺漏记入 `review.foreshadow_gap`,optimizer 据此补写。
- **风格一致性评估**:对一批章节跑风格打分,根据得分决定是否触发统一润色 optimizer。

### 设计要点

- 评估 prompt 必须是结构化输出,带 JSON schema,避免自然语言误导优化器。
- 优化器拿到的是结构化反馈,而非整章重写指令,避免“全部推翻重来”。
- 必须有最大回流次数,避免无限循环。

### 最小 demo / 伪代码

```python
from pydantic import BaseModel


class ChapterEvaluation(BaseModel):
    score: float              # 0.0 ~ 1.0
    needs_rewrite: bool
    issues: list[str]         # 结构化问题清单
    suggested_fix: str | None # 局部修改建议,不是整章重写


def evaluator(llm_eval, draft: str, context: dict) -> ChapterEvaluation:
    return EVAL_PROMPT | llm_eval.with_structured_output(ChapterEvaluation)


def optimizer(llm_writer, draft: str, eval_: ChapterEvaluation) -> str:
    if not eval_.needs_rewrite:
        return draft
    return llm_writer.invoke(
        {
            "draft": draft,
            "issues": eval_.issues,
            "fix": eval_.suggested_fix,
        }
    )


def eval_optimize_loop(llms, draft, context, max_retry=3):
    for _ in range(max_retry):
        eval_ = evaluator(llms.eval, draft, context)
        if not eval_.needs_rewrite:
            return draft
        draft = optimizer(llms.writer, draft, eval_)
    return draft
```

## 5. LangGraph 图状态机

### 落点

- **章节写作主图**(已在 doc 04):`plan → write → proofread → [history_check] → review_gate`,带条件边与回流边。
- **跨章节一致性扫描图**:逐章节点 + 冲突聚合节点 + 自动生成 `修订/跨审报告.md`。
- **批量章节写作图**:Supervisor 节点 + 多个写章节子图并行(Supervisor-Worker 也可改写成图)。

### 设计要点

- 每个节点返回 `state` 的 diff,不要修改共享内存。
- 节点的输入上下文由 `ContextPack` 裁剪,避免无谓 token 成本。
- `retry_count` 与 `interrupt_id` 必须在 state 中显式记录,便于可观测性。

### 最小 demo / 伪代码

```python
from typing import Literal
from langgraph.graph import END, StateGraph


class CrossAuditState(TypedDict):
    chapter_ids: list[str]
    per_chapter_issues: dict[str, list[str]]   # chapter_id -> issues
    conflicts: list[dict]
    report_path: str


def per_chapter_node(state: CrossAuditState) -> dict:
    out = {}
    for cid in state["chapter_ids"]:
        out[cid] = audit_chapter(cid)
    return {"per_chapter_issues": out}


def aggregate_node(state: CrossAuditState) -> dict:
    conflicts = find_cross_chapter_conflicts(state["per_chapter_issues"])
    report = render_report(conflicts)
    return {"conflicts": conflicts, "report_path": write_report(report)}


def build_cross_audit_graph():
    g = StateGraph(CrossAuditState)
    g.add_node("per_chapter", per_chapter_node)
    g.add_node("aggregate", aggregate_node)
    g.set_entry_point("per_chapter")
    g.add_edge("per_chapter", "aggregate")
    g.add_edge("aggregate", END)
    return g.compile()
```

## 6. Human-in-the-Loop(人在回路)

### 落点

- **题材与风格选择**:`/init` 后由用户确认题材定位、视角、人称、敏感等级。
- **大纲与目录确认**:每次生成大纲后 `confirm_action`,确认通过才允许 `/目录`。
- **关键决策点的反问**:`ask_user_choice`、`ask_user_text`,例如章节结束时由用户决定是否扩写番外。
- **回流干预**:审核失败后,通过 interrupt 把决策权交给用户,而不是机械回流重写。
- **敏感话题再确认**:涉及历史人物、现实题材、版权人物等高风险点必须由人确认。

### 设计要点

- HITL 不要做成“必填问卷”,只在决策不可逆或主观分歧大时触发,否则伤害体验。
- 中断后必须能恢复:LangGraph 的 `interrupt` 机制 + checkpoint 必须配套。
- 用户的回答要落盘到项目里(`大纲/大纲.md` 的修订记录、`AGENT.md` 的当前决策),而不是只在内存中。

### 最小 demo / 伪代码

```python
from langgraph.checkpoint.memory import MemorySaver


def ask_user_choice_node(state):
    question = "本章结尾是开放式还是闭合式?"
    # 由图中断抛出,REPL 捕获后显示给用户,用户在 REPL 输入后恢复
    interrupt_payload = {"type": "choice", "options": ["open", "closed"]}
    raise Interrupt(question=question, payload=interrupt_payload)


def build_graph_with_hitl():
    g = StateGraph(WriterState)
    g.add_node("ask_end_style", ask_user_choice_node)
    g.add_node("write", write_node)
    g.add_edge("ask_end_style", "write")
    return g.compile(
        checkpointer=MemorySaver(),
        interrupt_before=["ask_end_style"],
    )
```

注意:LangGraph 的实际 HITL API 命名随版本变化(`Interrupt` 类、`interrupt_before/after` 参数),本伪代码强调“中断位置可声明、状态可恢复”这一思想。

## 7. Supervisor-Worker(分层多智能体)

### 落点

- **批量章节写作组长**:Supervisor 把 `章纲 → 章 N` 派给 Worker(每个 Worker 内嵌章节写作子图),完成情况汇报 Supervisor,Supervisor 推进到下一章或失败回退。
- **跨章节一致性扫描 Supervisor**:Supervisor 调度“每章独立 Worker + 聚合 Worker”,前者用 Parallelization,后者用 Evaluator-Optimizer。
- **写作助手套件集成**:对外把整套 Writer Agent 当成一个 Worker,对上接受用户级 Supervisor 的指挥(例如嵌入 IDE 写作插件)。

### 设计要点

- Supervisor 不直接写正文,只做任务分配、状态推进、错误兜底。
- Worker 之间的通信必须经 Supervisor,避免横向乱发消息。
- Supervisor 的状态独立于每个 Worker,Worker 完成后只需汇报结构化结果。

### 最小 demo / 伪代码

```python
from dataclasses import dataclass


@dataclass
class ChapterAssignment:
    chapter_id: str
    title: str
    summary: str


@dataclass
class ChapterResult:
    chapter_id: str
    draft: str
    eval: ChapterEvaluation
    ok: bool


def supervisor_plan(toc: list[ChapterAssignment]) -> list[ChapterAssignment]:
    # 把多章任务排序;MVP 可串行,后续可并行派发
    return sorted(toc, key=lambda c: c.chapter_id)


def worker_write_chapter(llms, assignment: ChapterAssignment) -> ChapterResult:
    draft = llms.writer.invoke(assignment)
    eval_ = evaluator(llms.eval, draft, ctx_for(assignment.chapter_id))
    return ChapterResult(
        chapter_id=assignment.chapter_id,
        draft=draft,
        eval=eval_,
        ok=not eval_.needs_rewrite,
    )


def supervisor_drive(llms, toc: list[ChapterAssignment]) -> list[ChapterResult]:
    plan = supervisor_plan(toc)
    results: list[ChapterResult] = []
    for assign in plan:
        result = worker_write_chapter(llms, assign)
        if not result.ok:
            # 失败回流:不重写整章,只把结构化反馈交给人/优化器
            fix = ask_user_or_optimizer(result)
            result = worker_write_chapter(llms, replace(assign, hint=fix))
        results.append(result)
    return results
```

## 模式之间的组合关系

实际项目中,这些模式不是孤立使用的。下表给出常见的组合路径:

| 场景 | 组合模式 |
| --- | --- |
| `/init` 一次性初始化 | Prompt Chaining → 末端 HITL 让用户确认 |
| 用户输入分发 | Routing(LLM 路由器) |
| 章节写作 | LangGraph → 内嵌 Evaluator-Optimizer → 命中敏感时 HITL |
| 批量章节 | Supervisor-Worker → 每个 Worker 用 LangGraph |
| 多顾问审核 | Parallelization → 末位 Aggregator 收敛 |
| 跨章节一致性 | Supervisor-Worker → 内部 Parallelization → Evaluator-Optimizer 反复迭代 |
| 大纲 → 章纲 → 章纲细化 | Prompt Chaining 各阶段内部 Routing 决定分支 |
| 长跑中途回退 | LangGraph + HITL 给出“接受 / 回流 / 重写”三选一 |

## 落地建议

- 先选择**主导模式**,再围绕它补足其他模式。例如章节写作主导是 LangGraph,内部自然嵌入 Evaluator-Optimizer;批量写作主导是 Supervisor-Worker,每个 Worker 内部可以是 LangGraph。
- HITL 是“保护层”,不是“日常层”。只在不可逆决策、主观分歧、高风险话题处插入。
- Parallelization 必须在“彼此独立”的前提下使用,聚合阶段要走确定性合并。
- Routing 决策要落到 Pydantic 结构化输出,不要让 LLM 自由发挥。
- Superviser 和 Worker 的边界必须明确,Supervisor 不写正文,Worker 不改项目状态机。
- 在每个模式里都能写出最小可演示 demo,而不是把多种模式“糅合”到一段难以拆解的方法里。

## 验收标准

- 任何 LLM 调用都不应是裸 prompt,要么属于一个 Prompt Chaining 步骤,要么属于某个 LangGraph 节点。
- 任何用户输入都必须经过 Routing 决策,不能直接走向业务逻辑。
- 任何长任务都必须有 Evaluator-Optimizer 闭环或显式的人工回退路径。
- 任何独立子任务都应评估是否可并行;并行任务不能写共享可变状态。
- HITL 触发点必须可解释:为什么这个决策需要人介入。
- Supervisor-Worker 中,Supervisor 与 Worker 状态分离,Worker 完成情况以结构化结果汇报。

## 附录:七种模式速查

| 模式 | 一句话 | 适合本项目的特征 |
| --- | --- | --- |
| Prompt Chaining | 多个 LLM 调用按确定性代码串联 | 步骤固定、前后依赖、可单步缓存 |
| Routing | 由模型/规则决定走向 | 输入意图多样、状态机驱动 |
| Parallelization | 多个任务并发执行后聚合 | 任务彼此独立、需要缩短时延 |
| Evaluator-Optimizer | 评估 → 优化,迭代收敛 | 需要稳定质量、长任务可反复改进 |
| LangGraph 图状态机 | 显式图节点与条件边 | 流程复杂、需要可观测与 checkpoint |
| Human-in-the-Loop | 关键决策交给人 | 不可逆、主观分歧、高风险 |
| Supervisor-Worker | 1 个 Supervisor + 多个 Worker | 任务可分解、跨多子任务、需要统一协调 |
