# 检查点恢复与可观测性

## 业务背景

`/写作`、`/审核` 都是长任务。用户可能 Ctrl+C 中断,网络可能失败,LLM 可能超时或输出截断。CLI 需要让用户知道任务进行到哪里,也要能从安全位置恢复。

## 技术难点

长任务跨多个节点和文件写入,失败点很多。没有检查点就只能重跑,成本高且可能产生重复章节。没有可观测性,用户也无法判断是模型慢、检索慢、工具失败还是审核回流。

## 解决方案

使用 LangGraph checkpoint 记录节点级状态。每个节点结束后写入 SQLite checkpoint,用户中断时强制 flush 当前状态。CLI 启动时检测未完成会话,提示是否继续。

同时建立结构化日志和任务时间线:

- 节点开始/结束时间。
- 模型档位和 token 用量。
- RAG top-up 命中情况。
- 重试次数和回流原因。
- 工具调用结果。

生产日志写入 `~/.writer/logs/<date>.log`,开发模式输出更详细 debug。

## 最小 demo / 伪代码

```python
from pathlib import Path
import json
import time


class CheckpointStore:
    def __init__(self, path: Path) -> None:
        self.path = path

    def save(self, run_id: str, node: str, state: dict) -> None:
        payload = {
            "run_id": run_id,
            "node": node,
            "saved_at": time.time(),
            "state": state,
        }
        self.path.write_text(json.dumps(payload, ensure_ascii=False), encoding="utf-8")

    def load(self) -> dict | None:
        if not self.path.exists():
            return None
        return json.loads(self.path.read_text(encoding="utf-8"))


def run_node_with_checkpoint(store: CheckpointStore, run_id: str, node_name: str, state: dict) -> dict:
    new_state = run_node(node_name, state)
    store.save(run_id=run_id, node=node_name, state=new_state)
    log_event("node_finished", node=node_name, chapter=state.get("chapter_id"))
    return new_state
```

## 核心依赖版最小代码

```python
from typing import TypedDict

from langgraph.checkpoint.memory import MemorySaver
from langgraph.graph import END, StateGraph
from rich.console import Console


class RuntimeState(TypedDict):
    chapter_id: str
    draft: str


def write_node(state: RuntimeState) -> dict:
    return {"draft": f"{state['chapter_id']} 的章节草稿"}


checkpointer = MemorySaver()
graph = StateGraph(RuntimeState)
graph.add_node("write", write_node)
graph.set_entry_point("write")
graph.add_edge("write", END)

app = graph.compile(checkpointer=checkpointer)


def run_with_checkpoint(chapter_id: str) -> None:
    console = Console()
    config = {"configurable": {"thread_id": f"chapter:{chapter_id}"}}
    result = app.invoke({"chapter_id": chapter_id, "draft": ""}, config=config)
    console.print(result)
```

## 落地建议

- checkpoint 与项目建立稳定映射,避免多项目互相覆盖。
- 每个节点副作用尽量在节点成功后提交,失败时可回滚到进入节点前。
- Rich 输出中展示简短任务时间线,不要把 debug 信息刷满屏幕。
- 定义统一错误码:如 `LLM_TIMEOUT`、`LLM_OUTPUT_TRUNCATED`、`CHECKPOINT_CORRUPTED`、`TOOL_EXEC_FAILED`。

## 验收标准

- `/写作` 中途 Ctrl+C 后,下次启动能提示从哪个节点恢复。
- checkpoint 损坏时能备份旧库并新开会话。
- 章节末尾能汇总耗时、token、模型档位、重试次数。
- 日志中可以追踪一次章节生成经过了哪些节点。
