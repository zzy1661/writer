# 历史题材史实校验

## 业务背景

项目支持历史、架空历史、半架空历史等题材。历史题材需要创建 `史实/` 目录,并在写作和审核时由历史顾问检查人物、事件、年号、官职、时间线等内容。

## 技术难点

历史小说允许虚构,但关键史实不能随意篡改。难点不是简单判断“对或错”,而是要区分史实、合理演义、明确虚构和严重冲突。历史顾问如果常驻所有项目又会浪费 token;如果只靠通用 LLM 记忆,事实可靠性不足。

## 解决方案

在 `/init` 阶段由 GenreLocator 判断是否历史相关。只有历史题材才创建 `史实/` 并启用历史顾问条件边。史实校验采用本地史实资料优先:

- `史实/年表.md`:关键年份与事件。
- `史实/人物.md`:历史人物生卒、关系、命运。
- `史实/事件.md`:重大事件的发生顺序。
- `史实/考证.md`:用户补充资料和争议说明。

历史顾问输出要分级:硬冲突、可演义、需标注虚构、无问题。

## 最小 demo / 伪代码

```python
from typing import Literal, TypedDict


class HistoryIssue(TypedDict):
    level: Literal["hard_conflict", "fiction_ok", "needs_note", "ok"]
    text_claim: str
    history_anchor: str
    suggestion: str


def should_enable_historian(agent_state: dict) -> bool:
    return agent_state.get("题材", {}).get("是否历史题材", False)


def history_check(chapter_text: str, chapter_id: str) -> list[HistoryIssue]:
    history_chunks = rag_query(index="history", query=chapter_text, k=5)
    prompt = render_prompt(
        role="historian",
        chapter_text=chapter_text,
        history_chunks=history_chunks,
    )
    return call_llm_json(prompt)


def route_after_proofread(state: dict) -> str:
    if state["is_historical"]:
        return "history_check"
    return "review_gate"
```

## 核心依赖版最小代码

```python
from typing import Literal

from langchain_core.prompts import ChatPromptTemplate
from langchain_openai import ChatOpenAI
from pydantic import BaseModel


class HistoryCheckItem(BaseModel):
    level: Literal["hard_conflict", "fiction_ok", "needs_note", "ok"]
    text_claim: str
    history_anchor: str
    suggestion: str


class HistoryCheckReport(BaseModel):
    issues: list[HistoryCheckItem]


history_prompt = ChatPromptTemplate.from_messages(
    [
        ("system", "你是历史顾问。请基于史实资料检查正文,区分硬冲突、可演义、需标注虚构和无问题。"),
        ("human", "史实资料:\n{history}\n\n章节正文:\n{chapter_text}"),
    ]
)


def history_check_with_langchain(history: str, chapter_text: str) -> HistoryCheckReport:
    llm = ChatOpenAI(model="deepseek-v3", temperature=0)
    chain = history_prompt | llm.with_structured_output(HistoryCheckReport)
    return chain.invoke({"history": history, "chapter_text": chapter_text})
```

## 落地建议

- `is_historical` 写入 `AGENT.md`,启动时据此决定是否注册历史顾问。
- 历史检查节点只读取与当前章节相关的史实 chunk,避免全量塞入。
- 审核报告中史实问题必须给出“正文说法”和“史实锚点”。
- 对半架空题材提供“可虚构边界”字段,允许用户定义不可改动的历史底线。

## 验收标准

- 非历史项目不会创建或读取 `史实/`。
- 历史项目写到真实人物关键命运时能触发史实检查。
- 审核报告能区分”严重史实错误”和”可接受演义”。
- 用户可通过 `/史实` 查询本项目已登记史料。

## 2026-07-09 补充：chg-remove-rag / chg-project-skills 落地后的连带影响

- 历史题材的"正典上下文"原本设想走 `RagRetriever.query_history(chapter_id, k=5)` 召回(本文档早期方案)。`chg-remove-rag` 落地后,**RAG 已经删除**,历史题材与其它题材走相同的 `_build_canon_block` 4 层文件拼装:
  - `outline/大纲.md` 全文
  - `characters/*.md` 全文
  - `chapter_summaries.json` 切片
  - `manuscript/<latest>.md` 最近一章原文
- `史实/年表.md` 等文档仍然由 `create_workspace` 按题材 scaffold,但**不在** `_build_canon_block` 自动拼接范围内——LLM 工具循环按需 `safe_read_file("史实/年表.md")` 自行消费。换言之:历史题材正典是 LLM 主动读,引擎不再额外喂。
- `HistoryAgent.prompt` 的"史实资料优先"思路还在,但底层的"史实 chunk 召回"被 `project_search(query=chapter_text, path="史实/", ...)` + LLM 自己消费替代。
