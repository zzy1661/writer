# 伏笔生命周期与跨章节一致性

> **2026-07-08 重要修订**:本文档原描述的 `伏笔/伏笔库.md` Markdown 文件 + `ForeshadowItem` dataclass 形态已被**整体替换**为 YAML 结构化 ledger([OpenSpec `chg-remove-rag`](../../openspec/changes/archive/2026-07-08-chg-remove-rag/) + `openspec/specs/foreshadow-ledger/spec.md`)。原 `foreshadow_query(query: str)` 模糊查也改为 `foreshadow_search(id / tags / status / chapter_range / keyword)` 结构化查。
>
> 本文以**新形态**重写。删除 RAG 之前的伏笔管理流程保留为"历史背景"留档。

## 业务背景

长篇小说需要持续管理伏笔。每条伏笔有唯一 ID、状态、埋设章节、回收章节、关联标签和备注,写作时按章节查询,审核时检查是否按计划回收。

## 技术难点

伏笔是跨章节状态,不是单次文本生成问题。它可能经历"埋设、潜伏、回收"等阶段。若只靠 LLM 临场记忆,容易出现忘记回收、重复回收、提前揭示。

## 当前解决方案(2026-07-08 起)

### 1. 结构化 ledger: `<project_root>/伏笔.yaml`

**Schema**(per `openspec/specs/foreshadow-ledger/spec.md`):

```yaml
foreshadows:
  - id: F001
    tags: [主线, 身世]
    status: laid       # laid | paid
    laid_chapter: 3
    paid_chapter: 18    # null 表示尚未回收
    notes: 玉簪真实来历
  - id: F002
    tags: [伏笔, 江湖]
    status: paid
    laid_chapter: 5
    paid_chapter: 22
    notes: 神秘信件
```

**字段约束**(`writer/tools/builtin/foreshadow_ledger.py::_REQUIRED_FIELDS`):

- `id: str` — 唯一标识,建议 `F\d+`(`_ID_PATTERN` 校验)
- `tags: list[str]` — 多标签
- `status: "laid" | "paid"` — 仅两种
- `laid_chapter: int` — 埋设章节
- `paid_chapter: int | null` — 回收章节,null 表示未回收
- `notes: str` — 备注

任何字段缺失或类型错误 → `ForeshadowLedgerSchemaError`(domain exception),工具层 try/except 转 `ToolResult(metadata.error="schema")`,**不外溢到 engine 边界**。

### 2. 查询 API:`foreshadow_search` Tool

5 个 kw-only 参数,**全部 AND 组合**:

| 参数 | 类型 | 语义 |
| --- | --- | --- |
| `id` | `str \| None` | 精确匹配 `F\d+` |
| `tags` | `list[str] \| None` | 命中至少一个 tag(OR 内部 / AND 跨参数) |
| `status` | `"laid" / "paid" / "all"` | laid = 未回收;paid = 已回收;默认 `all` |
| `chapter_range` | `tuple[int, int] \| None` | `laid_chapter` 在 `[lo, hi]` 区间 |
| `keyword` | `str \| None` | 子串匹配 `id` / `notes` / `tags`(大小写敏感) |

实现位置:`writer/tools/builtin/foreshadow_tools.py::ForeshadowSearch` + `foreshadow_ledger.py::query_ledger`(纯函数,无 IO)。

**S0 路径**:`runtime.project_root` 是 sentinel `"/__no_project__"` 时,工具直接返回 `ToolResult(metadata.error="no_project_root")`,**不读盘**。

**rule 路由**:`RuleBasedIntentRouter` 在自然语言含"伏笔"或 `F\d+` 模式时,自动调 `_parse_foreshadow_args` 提取 `id` + `keyword` 双发:

```python
def _parse_foreshadow_args(text: str) -> dict[str, Any]:
    match = _FID_PATTERN.search(text)
    if match is None:
        return {"keyword": text.strip()}
    return {"id": match.group(0), "keyword": text.strip()}
```

### 3. Markdown 渲染(可选)

`伏笔/伏笔库.md` 当前**不是** ledger 主存储;YAML 才是。Markdown 渲染可由 LLM 工具循环按需用 `safe_read_file("伏笔.yaml")` + `safe_write_file("伏笔/伏笔库.md", ...)` 生成,纯文本展示。

### 4. ledger 写入路径

当前**未实装**专门的写入 Tool;LLM 工具循环直接 `safe_write_file` 写 `伏笔.yaml`。schema 校验由 `load_ledger` 提供(读时校验);`safe_write_file` 不带 schema 校验,LLM 需要按 SKILL.md directive 指引填字段(原 `/续写` `/改` 占位 SKILL.md 已 2026-07-09 删除)。

## 落地建议

- **不再**为 ledger 单独写 Python Tool(`foreshadow_update` 等);LLM 工具循环 + SKILL.md directive 教学够用
- ID 生成由 SKILL.md frontmatter 约定(`F\d+` 自增),LLM 在写 YAML 前先 `safe_read_file` 现有 ledger 算下一个 ID
- ledger 校验失败时 `ForeshadowLedgerSchemaError` 不应被 SKILL.md directive body 直接 raise;让 `safe_write_file` 写,读时由 `foreshadow_search` 触发 schema 错误提示
- 章节定稿后,SKILL.md `/大纲` `/目录` 提示 LLM 检查本章涉及的伏笔是否需要更新 `paid_chapter` / `status`(原 `/续写` `/改` 占位 SKILL.md 已 2026-07-09 删除)

## 验收标准

- ledger 写错字段名 → `foreshadow_search` 返回 `metadata.error="schema"`,REPL turn 不被中断
- 多条件(`tags` + `status` + `chapter_range` + `keyword`)查询必须全部命中(AND),不能 OR
- S0 路径下 `foreshadow_search` 不允许读盘,直接返回 sentinel 提示
- 自然语言"查一下 F003 玉簪"应自动路由到 `foreshadow_search(id="F003", keyword="玉簪")`
- ledger 文件位置是 `<project_root>/伏笔.yaml`,不是 `伏笔/伏笔.yaml`
