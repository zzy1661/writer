## `/骨架` 命令方案

### 1. 目标与非目标

**目标**：在已有 `大纲/大纲.md` + `大纲/章节目录.md` 上，为全书（或指定卷/区间）生成每章「开头 + 结尾」骨架，落盘后作为 `/创作` 的边界约束。

**非目标**：

- 不写章中正文、不跑完整 `write_chapter` 图
- 不替代 `/大纲` / `/目录`
- 不做纯 SKILL.md ReAct（步数/调度不够）

---

### 2. 产品形态

| 项         | 约定                                                              |
| ---------- | ----------------------------------------------------------------- |
| 命令       | `/骨架`（无英文别名，与现有文档一致）                             |
| 路由       | `start_workflow` → `skeleton_chapters`                            |
| 实现       | LangGraph 轻量图（Supervisor + 分批 Worker），**不是** skill      |
| 可选文档面 | 可放薄 `SKILL.md` 只做 `/帮助` 说明；执行仍走 workflow            |
| 可用状态   | 最低 **S4**（章节目录已生成）；S0-S3 拒绝并提示先 `/目录`         |

**语法（建议，与既有 `chapter_id="1.1"` 双层约定对齐）**：

```text
/骨架                         # 全书（按目录全量，mode=full）
/骨架 卷一                    # 仅指定卷，mode=volume, volume="卷一"
/骨架 1.1-1.20                # 卷一第 1-20 章区间，mode=range
/骨架 1.1-2.20                # 跨卷区间，mode=range
/骨架 rewrite                 # 覆盖已有骨架，rewrite=True, mode=full
/骨架 rewrite 卷二            # rewrite=True, mode=volume, volume="卷二"
/骨架 view                    # 只读预览摘要，不写盘，短路返回 chunks
/骨架 continue                # 从进度文件续跑，continue=True
```

**注**：`/骨架 1-20` 单层区间**不接受**——章节 ID 必须是双层 `卷·章`（与 `WriteChapterArgs.chapter_id="1.1"` 对齐，per `workflows/params.py:32`）。

---

### 3. 编排选型

```text
Routing(/骨架)
  → load_inputs（读 AGENT + 大纲 + 目录；缺则 failed）
  → parse_toc（目录 → ChapterTask[]）
  → plan_batches（按卷/固定批大小切分；目录即计划，不另起全书 P&E）
  → for batch in batches:                # Supervisor
        for chapter in batch:            # 批内串行（保衔接，prev_closing 串递）
            generate_open_close          # 单次 prose_client 调用
            persist_chapter_skeleton     # 直接 Path.write_text 写 骨架/<卷>/第N章.md
            update_progress              # 同步追加 骨架/进度.json
        （批间可后续再开并行）
  → finalize（写 骨架/索引.md；返回 WorkflowResult）
```

- **外层**：Supervisor-Worker（目录驱动）
- **局部**：可选「本卷情绪节拍表」一小段 P&E；默认 MVP 可跳过
- **单章**：单次 LLM（`deps.prose_client.generate_text`），不做 5 节点写作图
- **进度反馈**：**不**走 runner 流式 TextChunk（runner 单次消费 `WorkflowResult.chunks`，中途无法 yield——见 `runner/runner.py::_run_workflow`）。改为单章生成后**同步落盘** `骨架/进度.json`，最终 `WorkflowResult` 一次性给摘要 chunks + `artifacts["progress_path"]`，REPL 据此渲染进度条/计数
- **质控**：MVP 不做章章 Evaluator；可选卷末抽检留到 v2

### 4. 前置依赖与失败语义

| 条件                                | 行为                                                                                                |
| ----------------------------------- | --------------------------------------------------------------------------------------------------- |
| 无项目根                            | `failed` + 提示先 `writer new` / 进项目                                                             |
| 项目状态 `< S4`                     | `failed` + 提示先 `/目录`（章节目录是 `/骨架` 前置）                                                |
| 无 `大纲/大纲.md`                   | `failed` + 提示 `/大纲`                                                                             |
| 无 `大纲/章节目录.md`               | `failed` + 提示 `/目录`                                                                             |
| 已有骨架且无 `rewrite`              | `failed` + `metrics["conflict_path"]` 提示 `/骨架 rewrite` / `view`                                 |
| `view`                              | `completed`，只产出摘要 chunks，不写盘                                                              |
| `prose_client.name == "deterministic"` | `failed` + 提示配 `WRITER_API_KEY`（与 `_plan_chapter_node` 同款 strict raise，不退化模板）     |

### 5. 产物约定

**目录**（建议，与 `草稿/` 并列、不污染正文）：

```text
骨架/
  进度.json                  # 续跑用（与 chapter_summaries.json 风格一致，不引入 PyYAML）
  索引.md                    # 全书开/收一行摘要表
  卷一/
    第001章.md                # chapter_id=1.1
    第002章.md                # chapter_id=1.2
    ...
```

**写盘路径**：workflow 内**直接**用 `Path.write_text` 写入 `骨架/<卷>/第N章.md` 与 `骨架/进度.json`（与 `write_chapter._persist_outputs_node` 写 `草稿/chapter-{chapter_id}.md` 同款，per `write_chapter.py:322-326`）。**不**经 `safe_write_file` 工具，因此**不**需要扩 `DEFAULT_WRITE_WHITELIST`（per `tools/runtime.py:18-41`，当前不含 `骨架`，但 workflow 内直写绕过 runtime 白名单）。若未来要让 LLM-driven tool loop 写骨架文件，再扩白名单。

**单章文件模板**（章节 ID 用既有双层 `1.1` 形式）：

```markdown
# 第 1.1 章 · <目录标题>

## 元信息

- chapter_id: 1.1
- volume: 卷一
- 目录摘要: <从章节目录摘一句>

## 开头

<200–400 字：场景切入、人物在场、本章初始冲突钩子>
（禁止写完整高潮；禁止泄终局）

## 结尾

<150–300 字：本章收束势 + 通向下一章的钩子>
（须能接下章开头；本卷末章可收卷不硬留钩）

## 衔接备注

- 承接上章: <一句话>
- 交给下章: <一句话>
```

**字数常量**（与 `REVIEW_THRESHOLD = 7` 同款模块顶部常量，per `write_chapter.py:73`）：

```python
# writer/workflows/skeleton_chapters.py
OPEN_MAX_CHARS = 400
CLOSE_MAX_CHARS = 300
```

按题材微调留到后续 PR。

**进度文件 `骨架/进度.json`**：

```json
{
  "status": "running" | "completed" | "failed",
  "mode": "full" | "volume" | "range",
  "rewrite": false,
  "completed": ["1.1", "1.2"],
  "current": "1.3",
  "updated_at": "2026-07-17T10:30:00Z",
  "error": null
}
```

**注意**：`status` 与 `WorkflowResult.status` 是两个独立字段。`进度.json.status` 表达「本次运行落盘状态」；`WorkflowResult.status` 表达「本次运行最终退出」。

### 6. 图节点设计（MVP）

对齐现有 `write_chapter` / `review_chapter` 风格 + `_set_deps / _get_deps()` 模块级 deps 注入（per `write_chapter.py:367-396`）：

| 节点                    | 职责                                                                                                                       |
| ----------------------- | -------------------------------------------------------------------------------------------------------------------------- |
| `load_inputs`           | 读 AGENT（题材/架构方法，复用 `read_genre_from_agent` / `read_architecture_method_from_agent`，per MEMORY 2026-07-16）；读大纲；读目录；解析 `SkeletonArgs`；检查 `project_state >= S4` |
| `parse_toc`             | TOC → `list[ChapterTask]`；过滤卷/区间；处理 rewrite                                                                      |
| `init_or_load_progress` | 新建或读 `骨架/进度.json`；`continue` 跳过已完成                                                                          |
| `generate_batch`        | 循环：拼 prompt（大纲切片 + 本卷上下文 + **上章结尾** + 本章目录行 + 上一 chapter_id 的 prev_closing）→ `deps.prose_client.generate_text` → 校验开/收非空 → 调 `persist_skeleton` |
| `persist_skeleton`      | **直接** `Path.write_text` 写 `骨架/.../第N章.md`；**直接**追加 `骨架/进度.json`；返回 metric `chapter_count` 累计         |
| `finalize`              | 写 `骨架/索引.md`；`WorkflowResult(status="completed", artifacts=..., metrics={chapter_count, mode, rewrite, resumed})`    |

条件边：

- 任务为空 → `finalize`（或 `failed`「无匹配章节」）
- 生成异常 → **写 `进度.json` 为 `failed`**，返回 `WorkflowResult(status="failed", metrics={"partial_chapters": N, "progress_path": "..."})` → runner 映射为 `Done(aborted, payload={"error": ..., "metrics": ...})`
- 批未完成 → 自环 `generate_batch`（或外层 for，图内用状态机计数）

**部分完成的中断/续跑语义**：用 `status="failed"` + `metrics={"partial_chapters": N, "progress_path": "..."}` 表达。**不**用 `status="pending"`——runner 当前把 `pending` 专用于 `needs_rewrite` 信号（per `runner/runner.py:392-407` + `runner/events.py:24` 注释），滥用会让 REPL 误判为"需要重写"。

**状态字段（TypedDict 草案）**：

```python
class SkeletonState(TypedDict, total=False):
    project_root: str
    args: dict          # mode / volume / start / end / rewrite / continue / view
    tasks: list[dict]   # chapter_id, title, volume, toc_blurb
    cursor: int
    prev_closing: str
    completed_ids: list[str]
    artifacts: dict
    metrics: dict
    error: str
```

### 7. 单章 Prompt 要点

输入块：

1. 题材 + 架构方法（AGENT，行 2 / 行 3，**复用** `state.py::read_genre_from_agent` + `read_architecture_method_from_agent`，与 `/大纲` SKILL.md 同源）
2. 全书一句话核心 / 本卷定位（从大纲截）
3. 本章目录标题 + 前后各 1 章目录行（局部上下文）
4. **上章「结尾」全文**（首章则用卷首设定 / 大纲卷摘要）
5. 约束：只写 `## 开头` / `## 结尾`；不写中间；字数上限 `OPEN_MAX_CHARS=400` / `CLOSE_MAX_CHARS=300`；语气贴题材

输出：强制 Markdown 两段 `## 开头` + `## 结尾`，节点侧正则/简单解析失败则重试 1 次，再失败 → 写 `进度.json.status="failed"` + `WorkflowResult(status="failed")` 中止（**不**留空洞文件）。

### 11. Done / 观测

对齐 `WorkflowResult` 3 状态映射（per `runner/runner.py::_run_workflow` + `workflows/types.py:22`）：

- **成功**：`status="completed"` → `Done(reason="workflow_completed")`
  - `artifacts`: `skeleton_root`, `index_path`, `progress_path`
  - `metrics`: `chapter_count`, `mode`, `volume`, `rewrite`, `resumed`
- **完全失败**：`status="failed"` → `Done(reason="aborted", error=...)`（如确定性模式 / 无项目根 / 无目录）
- **部分完成被中断**：`status="failed"` → `Done(reason="aborted", error=..., metrics={"partial_chapters": N, "progress_path": "..."})` → REPL 提示「已写 N 章，请 `/骨架 continue`」

**关键**：**不**使用 `status="pending"` 表达部分完成（runner 当前把 `pending` 专用于 `needs_rewrite` 单义，per `runner/runner.py:392-407`）。如未来需要区分「自然中断」与「真失败」，在 `metrics["decision"]` 上扩枚举（如 `"interrupted"`），不动 `status`。

CLI：长操作 spinner（文档已点名 `/骨架`）；最终摘要从 `progress_path` 渲染。

### 13. 验收标准（阶段二对齐）

1. 有大纲+目录时，`/骨架` 能为目录中每章生成含「开头」「结尾」的文件
2. 第 N 章开头能承接第 N−1 章结尾（抽检相邻 3 对）
3. 无目录 / S3 状态时明确失败，不写半套文件
4. `rewrite` 可覆盖；默认不覆盖
5. 不经过 skill 的 5 步 ReAct；走 `workflow_completed`
6. 帮助与命令流程文档中 `/骨架` 从「占位」变为「已实装」
7. **新增（PR1）**：deterministic 模式下 `/骨架` raise，提示配 `WRITER_API_KEY`，**不**退化模板（与 `_plan_chapter_node` 同款契约）
8. **新增（PR1）**：`/骨架 view` 短路返回 chunks，**不**落盘任何文件
9. **新增（PR2）**：中断后续跑只补差量（写过的 chapter_id 跳过）
10. **新增（PR3）**：`_build_canon_block` 包含 `骨架/*.md`，`_draft_chapter_node` system prompt 注入开/收约束
